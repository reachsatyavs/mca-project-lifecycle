Semester 1 — Foundation

Design — Technical Design Document

Example Project: Online Food Delivery System

The Technical Design Document (TDD) translates requirements into a technical blueprint. It defines how the system will be built — architecture, database, class structure, and technology choices.

📖 What is System / Technical Design?

System Design is the process of deciding how software will be built — before any code is written. It answers questions like:

  • Architecture — How many tiers? Monolith or microservices? Which cloud?
  • Data Model (ER Diagram) — What data do we store, and how is it structured in the database?
  • Object Design (Class Diagram) — What are the main objects (classes) in code, and how do they relate?
  • API Design — What endpoints will the backend expose to the frontend?
  • Technology Stack — Which programming language, framework, and database?

The output of this phase is the Technical Design Document (TDD) — a living document reviewed and approved before coding begins. A good TDD prevents costly rework in later semesters.

🤔 Students Often Confuse These Three — Here is the Difference

All three diagrams live in your TDD, but each answers a completely different question:

1. System Architecture

The high-level structure of the Online Food Delivery System

The recommended architecture is a 3-tier web application with clear separation between Presentation, Application Logic, and Data layers — this is the industry-standard approach for web systems.

Architecture Tiers

TierComponentTechnologyResponsibility
Presentation (Frontend)Web Browser SPA or MPAReact.js / HTML+BootstrapRenders UI, collects user input, calls backend APIs
Application (Backend)REST API ServerExpress.js (Node.js) / Spring Boot (Java) / Django (Python)Business logic, authentication, order processing, API endpoints
Data (Database)Relational DBPostgreSQL / MySQLStores all persistent data: users, restaurants, orders, menus

Additional Components

  • Payment Gateway: Razorpay / Stripe Sandbox — external service called by backend for payment processing.
  • Email Service: SendGrid / Gmail SMTP — sends order confirmations, status updates, password reset emails.
  • File Storage: Cloudinary / local storage — stores restaurant and food item images.
  • Cache (Semester 2 onwards): Redis — caches frequently accessed data like restaurant lists and menus.

Deployment Architecture

Browser ──HTTPS──► [Render / Railway] │ ┌──────┴──────┐ │ Web Server │ (Gunicorn / Nginx) │ (Backend) │ └──────┬──────┘ │ SQL queries ┌──────▼──────┐ │ PostgreSQL │ (Supabase / Railway DB) └─────────────┘ │ ┌────────────┼────────────┐ │ │ │ Razorpay SendGrid Cloudinary (Payments) (Email) (Images)
Why 3-Tier?

Separating concerns means you can change the frontend without touching the backend, and vice versa. In your project, the restaurant owner dashboard and the customer portal are both separate frontends talking to the same backend API — this is only clean because you have a proper API layer.

2. ER Diagrams

Entity-Relationship model for the Food Delivery database

📖 What is an ER Diagram?

An Entity-Relationship (ER) Diagram is a visual blueprint of a database. It shows:

  • Entities — the "things" your system stores data about (e.g., User, Order, Restaurant). Drawn as rectangles.
  • Attributes — the properties of each entity (e.g., User has name, email, phone). Drawn inside the rectangle.
  • Relationships — how entities are connected (e.g., a User places many Orders). Drawn as lines with cardinality symbols.
  • Cardinality — how many instances of one entity relate to another: One-to-One (1:1), One-to-Many (1:M), Many-to-Many (M:N).

An ER diagram is created before any database tables are created. It is the design; the SQL schema is the implementation. Both must match.

Cardinality Notation (Crow's Foot)

We use Crow's Foot notation — the standard in industry tools like dbdiagram.io and draw.io:

Reading the Diagram
  • PK — Primary Key (unique identifier for each row)
  • FK — Foreign Key (references another table's PK)
  • UK — Unique Key (value must be unique, e.g., email)
  • ||──|| — Exactly one on both sides (1:1)
  • ||──o{ — One on left, zero-or-many on right (1:M)
  • ||──|{ — One on left, one-or-many on right (mandatory M)
  • ||──o| — One on left, zero-or-one on right (optional 1:1)

Corresponding SQL (for the core tables)

-- Users table (parent entity for all roles) CREATE TABLE users ( user_id SERIAL PRIMARY KEY, name VARCHAR(100) NOT NULL, email VARCHAR(255) UNIQUE NOT NULL, password_hash VARCHAR(255) NOT NULL, phone VARCHAR(15), role VARCHAR(20) CHECK (role IN ('CUSTOMER','OWNER','PARTNER','ADMIN')), created_at TIMESTAMP DEFAULT NOW() ); -- Address: one user → many addresses (1:M) CREATE TABLE addresses ( address_id SERIAL PRIMARY KEY, user_id INT NOT NULL REFERENCES users(user_id) ON DELETE CASCADE, street VARCHAR(255) NOT NULL, city VARCHAR(100) NOT NULL, pincode VARCHAR(10), lat DECIMAL(9,6), lng DECIMAL(9,6), is_default BOOLEAN DEFAULT FALSE ); -- Restaurant: one owner-user → many restaurants (1:M) CREATE TABLE restaurants ( restaurant_id SERIAL PRIMARY KEY, owner_id INT NOT NULL REFERENCES users(user_id), name VARCHAR(150) NOT NULL, cuisine_type VARCHAR(50), address TEXT, status VARCHAR(20) DEFAULT 'PENDING' CHECK (status IN ('PENDING','APPROVED','REJECTED')), avg_rating DECIMAL(3,2) DEFAULT 0.00 ); -- Order: one customer → many orders, one restaurant → many orders (M:1 both) CREATE TABLE orders ( order_id SERIAL PRIMARY KEY, customer_id INT NOT NULL REFERENCES users(user_id), restaurant_id INT NOT NULL REFERENCES restaurants(restaurant_id), partner_id INT REFERENCES users(user_id), -- nullable until assigned delivery_addr_id INT NOT NULL REFERENCES addresses(address_id), status VARCHAR(20) DEFAULT 'PLACED', subtotal DECIMAL(10,2) NOT NULL, delivery_fee DECIMAL(6,2) NOT NULL DEFAULT 40.00, total_amount DECIMAL(10,2) NOT NULL, created_at TIMESTAMP DEFAULT NOW() ); -- OrderItem: one order → many items, M items → many OrderItems (1:M + M:1) CREATE TABLE order_items ( order_item_id SERIAL PRIMARY KEY, order_id INT NOT NULL REFERENCES orders(order_id) ON DELETE CASCADE, menu_item_id INT NOT NULL REFERENCES menu_items(item_id), quantity INT NOT NULL CHECK (quantity > 0), unit_price DECIMAL(8,2) NOT NULL, subtotal DECIMAL(10,2) NOT NULL ); -- Payment: exactly one per order (1:1) CREATE TABLE payments ( payment_id SERIAL PRIMARY KEY, order_id INT UNIQUE NOT NULL REFERENCES orders(order_id), gateway_txn_id VARCHAR(100), amount DECIMAL(10,2) NOT NULL, method VARCHAR(20) CHECK (method IN ('ONLINE','COD')), status VARCHAR(20) DEFAULT 'PENDING', paid_at TIMESTAMP );

3. Class Diagrams

Object-oriented structure of the system's key domain classes

📖 What is a Class Diagram?

A Class Diagram (UML) shows the object-oriented structure of a software system. It describes:

  • Classes — blueprints for objects. Each class has a name, attributes (data), and methods (behaviour). Drawn as a 3-part rectangle: name / attributes / methods.
  • Visibility — + public, - private, # protected.
  • Inheritance (Generalisation) — a child class extends a parent class, inheriting all its attributes and methods. Drawn as a hollow arrow (△) from child to parent.
  • Association — one class uses another (e.g., Order has a Customer). Drawn as a plain line.
  • Composition — strong ownership; child cannot exist without parent (e.g., OrderItem is part of Order). Drawn as a filled diamond ◆.
  • Aggregation — weak ownership; child can exist independently. Drawn as hollow diamond ◇.

Inheritance Hierarchy: User System

All actors (Customer, Restaurant Owner, Delivery Partner, Admin) are types of users — they share common attributes and methods but have different specialised behaviour. This is classic inheritance. The hollow triangle arrow (△) means "extends / inherits from".

Why Inheritance Here?

BenefitExample
Code reuseLogin, logout, and password-reset logic is written once in BaseUser — all subclasses get it for free.
Single users tableOne users DB table stores all roles. The role column distinguishes them. (Table-per-hierarchy pattern.)
PolymorphismAny function that accepts a BaseUser object works with Customer, Owner, and Partner alike — e.g., send_notification(user).
Override where neededget_profile() returns different data per role: Customer returns saved addresses; Partner returns vehicle info.

Composition: Order contains OrderItems

Composition means an OrderItem cannot exist without its parent Order. If the Order is deleted, all its OrderItems are deleted too (enforced via ON DELETE CASCADE in SQL).

State Machine: Order Status

PLACED ──► ACCEPTED ──► PREPARING ──► PICKED_UP ──► DELIVERED │ │ ▼ ▼ CANCELLED CANCELLED (auto after 5 min with no restaurant response) Valid transitions: PLACED → ACCEPTED (by restaurant) PLACED → CANCELLED (by customer OR auto-timeout) ACCEPTED → PREPARING (by restaurant) ACCEPTED → CANCELLED (by restaurant — rejection) PREPARING → PICKED_UP (by delivery partner) PICKED_UP → DELIVERED (by delivery partner) Invalid (enforced in code): DELIVERED → any other status ← FORBIDDEN CANCELLED → any other status ← FORBIDDEN

Model this as an enum with a VALID_TRANSITIONS map in your Order class. When transition_to(new_status) is called, it checks the map before updating — this enforces the FSM at the application layer, not just the database.
