1. Test Case Execution

Running the test cases written in Semester 2 against the full deployed system

Why Execute in Semester 3 and Not Semester 2?

In Semester 2 you build and unit-test individual modules. But formal test case execution needs the fully integrated system — all modules wired together, deployed on staging, with real data flowing through. That complete system is only ready at the start of Semester 3. Running TCs on partial code gives incomplete and misleading results.

Each execution row traces back to the Feature Registry (FOOD-xxx) and the Test Cases written in Semester 2 (TC-xx). This completes the chain: Feature Plan → Sprint → Test Case Written → Test Case Executed.

2. Alpha Testing

Internal testing by the development team before external users

Alpha testing is conducted by the team itself on the deployed staging environment. It is the first end-to-end test of the complete system — all modules integrated and running together.

Alpha Test Scope for Food Delivery

  • All 14 Use Cases from the SRS must be executed manually at least once.
  • All edge cases from business rules must be verified (min order, free delivery threshold, cancellation window, etc.).
  • All three portals (Customer, Restaurant, Delivery Partner) tested across Chrome, Firefox, and Safari.
  • Mobile responsiveness verified at 375px (iPhone SE) and 414px (iPhone Pro Max).
  • Payment flows: successful payment, failed payment, COD order — all verified with Razorpay sandbox.

3. System / End-to-End Tests

⭐ Good to Have

Automated browser-based tests of complete user journeys

⭐ Good to Have — Manual system testing (Section 2 Alpha) is mandatory. Automated E2E with Playwright is impressive and adds real value but is optional for MCA.

E2E tests use a browser automation tool (Playwright or Selenium) to simulate a real user clicking through the application. They catch integration bugs that unit tests cannot — like a broken redirect after payment or a missing cookie.

E2E Test: Complete Order Flow (Playwright)

// tests/e2e/order_flow.spec.ts (Playwright) import { test, expect } from '@playwright/test'; test('Customer places a complete order with COD', async ({ page }) => { // 1. Login await page.goto('/login'); await page.fill('[name=email]', 'test@customer.com'); await page.fill('[name=password]', 'Test@1234'); await page.click('button[type=submit]'); await expect(page).toHaveURL('/home'); // 2. Browse and select restaurant await page.click('text=Biryani House'); await expect(page).toHaveURL(/\/restaurants\/\d+/); // 3. Add item to cart await page.click('button:has-text("Add to Cart")'); await expect(page.locator('.cart-badge')).toHaveText('1'); // 4. Proceed to checkout await page.click('text=Checkout'); await page.selectOption('[name=address]', '1'); // Select saved address await page.click('text=Cash on Delivery'); await page.click('text=Place Order'); // 5. Verify order confirmation await expect(page.locator('.order-status')).toHaveText('Order Placed'); await expect(page.locator('.order-id')).toBeVisible(); });
Playwright vs Selenium

Playwright (by Microsoft) is newer, faster, and has better async support than Selenium. It works with JavaScript/TypeScript, Python, and Java. For a student project, Playwright with JavaScript is the recommended choice.

4. Performance Tests

⭐ Good to Have

Validating that the system meets NFR-1 (page load) and NFR-3 (concurrent users)

⭐ Good to Have — A load test with results screenshot in your report is impressive. Evaluators rarely expect full perf testing at MCA level.

Load Test Script (k6)

// tests/perf/restaurants.js import http from 'k6/http'; import { check, sleep } from 'k6'; export const options = { stages: [ { duration: '30s', target: 10 }, // ramp up to 10 virtual users { duration: '1m', target: 30 }, // stay at 30 users for 1 minute { duration: '20s', target: 0 }, // ramp down ], thresholds: { http_req_duration: ['p(95)<500'], // 95% requests under 500 ms http_req_failed: ['rate<0.01'], // less than 1% errors }, }; // Login once per VU and reuse the token export function setup() { const res = http.post('https://your-app.render.com/api/auth/login', JSON.stringify({ email: 'test@customer.com', password: 'Test@1234' }), { headers: { 'Content-Type': 'application/json' } }); return { token: JSON.parse(res.body).token }; } export default function (data) { const headers = { Authorization: `Bearer ${data.token}` }; http.get('https://your-app.render.com/api/restaurants', { headers }); check(http.get('https://your-app.render.com/api/restaurants/1/menu', { headers }), { 'menu status 200': r => r.status === 200, }); sleep(1); }

Install and Run

# macOS brew install k6 # Run the test k6 run tests/perf/restaurants.js # Sample output to paste in report: # http_req_duration p(95)=412ms ← under 500ms target ✓ # http_req_failed 0.00% ← 0 errors ✓

6. Beta / Peer Review Testing

⭐ Good to Have

Getting external users to test the system before go-live

⭐ Good to Have — Peer beta testing shows maturity in the testing process. If time is limited, a quick 2–3 person peer review is still valuable to mention in the report.

Beta testing gives the system to real users — people who weren't involved in building it. They will discover usability issues, edge cases, and bugs the team has become blind to.

Beta Testing Plan for Food Delivery

  • Beta Testers: 8–10 other MCA students (different batch or section).
  • Duration: 2 weeks of testing with daily feedback.
  • Test Scenarios Given to Beta Users:
    • "Register as a new customer and place your first order."
    • "As a restaurant owner, add 3 menu items and accept an incoming order."
    • "Try to place an order with an empty cart."
    • "Use the system on your mobile phone."
  • Feedback Collection: Google Form with fields: Bug/Suggestion, Screenshot, Severity, Steps to reproduce.
  • Exit Criteria: All P1 bugs fixed; P2 bugs assigned to sprint; feedback summary documented.
