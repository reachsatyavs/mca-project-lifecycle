Why Design Changes in Semester 3?

Adapting the design based on real-world feedback

The design created in Semesters 1–2 was based on theoretical requirements. Semester 3 testing and beta user feedback reveals mismatches between the original design and actual usage patterns.

Typical Design Revisions for Food Delivery

Issue FoundDesign ChangeWhere Updated
Beta users confused by checkout flow — too many stepsRedesign checkout to single-page with accordion (UX revision)Wireframes + Frontend code
Restaurant listing page slow (400ms) with 50 restaurantsAdd Redis caching for restaurant list with 5-minute TTLArchitecture diagram + TDD
Order history page loads all orders — slow for power usersAdd cursor-based pagination (not page numbers — better for real-time data)API design + Sequence diagrams
Delivery partner assignment timeout logs show many failuresChange assignment algorithm: try 3 km, then 5 km, then 7 km with increasing timeoutsTDD algorithm section
Update the TDD

Every design revision must be reflected in the TDD. If you change the database schema in Semester 3, create a new migration AND update the ER diagram in the TDD. Outdated documentation is worse than no documentation — it misleads the team.

2. FoodBot — Agentic AI: Architecture

⭐ Good to Have

System architecture for the local LLM-powered customer support agent

📖 What is an Agentic AI? (vs a Chatbot)
AspectChatbotAgentic AI (FoodBot)
How it worksPattern-match input → return pre-written replyLLM reasons about the message, decides which tool to call, calls it, reasons about the result, then replies
Data accessStatic FAQ text onlyQueries live DB: order status, reviews, revenue, partner location
ActionsNone — read onlyCan cancel orders, apply wallet credits, create support tickets, send notifications
ResponsesSame answer for everyonePersonalised — uses customer's actual order history and context
Patternif/else or keyword matchReAct loop: Reason → Act → Observe → Reason again

3. FoodBot — Class Diagram

Object-oriented structure of the AI agent components

%%{init: {'theme': 'dark'}}%% classDiagram class FoodBotAgent { +string model +string systemPrompt +OllamaClient ollama +Tool tools +chat(sessionId, userId, message) string +buildMessages(sessionId) Message +runReActLoop(messages) string } class OllamaClient { +string host +string model +chat(messages, tools) LLMResponse +listModels() string } class Tool { +string name +string description +object parameters +execute(params) ToolResult } class GetOrderStatusTool { +execute(orderId) ToolResult } class RecommendDishesTool { +execute(customerId) ToolResult } class ResolveIssueTool { +execute(orderId, issue) ToolResult } class OwnerAnalyticsTool { +execute(restaurantId, period) ToolResult } class EscalateToHumanTool { +execute(sessionId, summary) ToolResult } class ConversationSession { +int sessionId +int customerId +string status +datetime createdAt +getMessages() Message } class Message { +string role +string content +string toolName +string toolResult +datetime createdAt } FoodBotAgent --> OllamaClient : uses FoodBotAgent --> Tool : orchestrates FoodBotAgent --> ConversationSession : reads Tool <|-- GetOrderStatusTool : extends Tool <|-- RecommendDishesTool : extends Tool <|-- ResolveIssueTool : extends Tool <|-- OwnerAnalyticsTool : extends Tool <|-- EscalateToHumanTool : extends ConversationSession --> Message : contains
Design Decisions
DecisionRationale
Tool abstract base classAll tools share the same interface — execute(params) — so the agent can call any tool without knowing its internals (polymorphism)
FoodBotAgent owns the ReAct loopSingle responsibility — the agent orchestrates reasoning; tools handle actions; Ollama handles language
ConversationSession + Message persisted in DBMulti-turn memory — the agent knows what was said earlier in the conversation, enabling coherent dialogue
OllamaClient separate classSwappable — replace Ollama with OpenAI/Gemini by swapping just this class, not the agent logic

4. FoodBot — ER Diagram (New Tables)

New database entities required to support the AI agent

The AI agent adds 4 new tables to the existing schema. No existing tables are modified — the AI layer is purely additive, which means zero risk of breaking the core system.

erDiagram USER { int user_id PK string name string email string role } AI_CONVERSATION_SESSION { int session_id PK int customer_id FK string status string context_summary datetime created_at datetime closed_at } AI_MESSAGE { int message_id PK int session_id FK string role text content string tool_name text tool_result int tokens_used datetime created_at } SUPPORT_TICKET { int ticket_id PK int session_id FK int customer_id FK text issue_summary string status text resolution datetime created_at datetime resolved_at } WALLET_CREDIT { int credit_id PK int customer_id FK int order_id FK decimal amount string reason string applied_by datetime expires_at datetime created_at } ORDER { int order_id PK int customer_id FK string status decimal total_amount } USER ||--o{ AI_CONVERSATION_SESSION : "has" AI_CONVERSATION_SESSION ||--|{ AI_MESSAGE : "contains" AI_CONVERSATION_SESSION ||--o| SUPPORT_TICKET : "may create" USER ||--o{ SUPPORT_TICKET : "raised by" USER ||--o{ WALLET_CREDIT : "credited to" ORDER ||--o{ WALLET_CREDIT : "compensation for"

New Table Descriptions

TablePurposeKey Fields
ai_conversation_sessionOne row per chat session. Tracks whether a session is active, escalated, or closed. Stores a summary for long contexts.status: active / escalated / closed
context_summary: LLM-compressed summary of old messages
ai_messageEvery message in a session — user input, assistant replies, and tool call/result pairs. This is the agent's memory.role: user / assistant / tool
tool_name: which tool was called
tool_result: what the tool returned
support_ticketCreated by EscalateToHumanTool when the agent cannot resolve an issue. Human agent picks it up from the admin panel.status: open / in_progress / resolved
wallet_creditCompensation credits applied by ResolveIssueTool. Applied to customer's next order automatically.applied_by: "FoodBot-AI" or admin name
expires_at: credits expire after 30 days

5. FoodBot — Sequence Diagram

Component interactions for the Issue Resolution use case

This sequence shows UC-AI-3: Order Issue Resolution — the most complex agentic flow, where the LLM makes two sequential tool calls before generating a final response.

%%{init: {'theme': 'dark'}}%% sequenceDiagram autonumber actor C as Customer participant FE as Chat Widget participant API as Express API participant AG as FoodBotAgent participant OL as Ollama LLM participant DB as PostgreSQL C->>FE: "My food arrived cold and Garlic Naan was missing" FE->>+API: POST /api/ai/chat { sessionId, message } API->>+AG: invoke(sessionId, userId, message) AG->>DB: INSERT ai_message (role=user) AG->>DB: SELECT messages WHERE session_id=? (load history) DB-->>AG: conversation history (last 10 messages) AG->>+OL: chat(systemPrompt + history + toolDefs) Note over OL: Thinks: "User has a complaint about an order. I should get the order details first." OL-->>-AG: tool_call: get_order_status { orderId: 42 } AG->>DB: SELECT order + items + delivery timestamps WHERE order_id=42 DB-->>AG: { status: DELIVERED, delivered_at, late_by: 22min, items: [...] } AG->>DB: INSERT ai_message (role=tool, tool_name=get_order_status) AG->>+OL: chat(history + tool result) Note over OL: Thinks: "Order was 22 min late and Garlic Naan (₹80) is confirmed in the order. Customer deserves compensation." OL-->>-AG: tool_call: apply_wallet_credit { customerId: 7, amount: 120, reason: "late delivery + missing item" } AG->>DB: INSERT wallet_credits { amount: 120, applied_by: "FoodBot-AI" } DB-->>AG: credit_id: 15 confirmed AG->>DB: INSERT ai_message (role=tool, tool_name=apply_wallet_credit) AG->>+OL: chat(history + tool result) Note over OL: Both tools executed. Now generate a final empathetic response. OL-->>-AG: "We're really sorry about your experience! ₹120 has been added to your FoodWallet..." AG->>DB: INSERT ai_message (role=assistant, content=finalResponse) AG-->>-API: finalResponse text API-->>-FE: 200 OK { message: "We're really sorry..." } FE-->>C: Display AI response with credit confirmation
What Makes This Agentic — Not Just a Query
StepWhat the LLM decided (not hardcoded)
Step 7–8LLM chose to call get_order_status first — it could have asked the customer for the order ID instead, but inferred it from session context
Step 10–12LLM decided ₹120 (not ₹80 just for the missing item) — it reasoned that late delivery compounded the issue and compensated proportionally
Steps 13–16LLM chose NOT to escalate to human — it resolved the issue itself. A different complaint might trigger escalation
Final responsePersonalised message referencing the actual items and amounts — not a generic template
