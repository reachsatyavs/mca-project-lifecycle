Why Design Changes in Semester 3?

Adapting the design based on real-world feedback

The design created in Semesters 1–2 was based on theoretical requirements. Semester 3 testing and beta user feedback reveals mismatches between the original design and actual usage patterns.

Typical Design Revisions for Food Delivery

Issue FoundDesign ChangeWhere Updated
Beta users confused by checkout flow — too many stepsRedesign checkout to single-page with accordion (UX revision)Wireframes + Frontend code
Restaurant listing page slow (400ms) with 50 restaurantsAdd Redis caching for restaurant list with 5-minute TTLArchitecture diagram + TDD
Order history page loads all orders — slow for power usersAdd cursor-based pagination (not page numbers — better for real-time data)API design + Sequence diagrams
Delivery partner assignment timeout logs show many failuresChange assignment algorithm: try 3 km, then 5 km, then 7 km with increasing timeoutsTDD algorithm section
Update the TDD

Every design revision must be reflected in the TDD. If you change the database schema in Semester 3, create a new migration AND update the ER diagram in the TDD. Outdated documentation is worse than no documentation — it misleads the team.
