Semester 4 — Production

Coding — Bug Fixes & Patches

Example Project: Online Food Delivery System

In Semester 4, coding is focused on fixing production bugs and security patches — not building new features. Every change follows a disciplined hotfix process to minimise risk.

🔧

Production Bug Fix Process

A disciplined approach to fixing bugs in a live system

Fixing bugs in production is different from fixing bugs during development. In production, a bad fix can break a working system. Every fix must be validated on staging before deploying to production.

Hotfix Workflow

# 1. Create hotfix branch from main (NOT develop) git checkout main git checkout -b hotfix/fix-order-minimum-validation # 2. Make the minimal fix # Edit only the affected file(s) # Add/update the relevant unit test # 3. Run full test suite locally jest --tb=short # 4. Push and create PR to main AND develop git push origin hotfix/fix-order-minimum-validation # 5. Deploy hotfix branch to staging for verification # 6. Get code review approval # 7. Merge to main → triggers CI/CD to production # 8. Merge to develop → keeps develop branch in sync

The "Minimal Fix" Principle

In production, make the smallest possible change that fixes the bug. Don't refactor surrounding code. Don't add unrelated features. The larger the change, the higher the risk of introducing new bugs. Refactoring belongs in a regular sprint — not a hotfix.
