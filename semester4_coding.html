1. Production Bug Fix Process

A disciplined hotfix approach that keeps production stable

Fixing bugs in production is different from fixing bugs during development. In production, a bad fix can break a working system. Every fix must be validated on staging before deploying to production.

Hotfix Workflow

# 1. Create hotfix branch from main (NOT develop) git checkout main git checkout -b hotfix/fix-order-minimum-validation # 2. Make the minimal fix — edit only the affected file(s) # Add/update the relevant unit test # 3. Run full test suite locally npx jest --runInBand # 4. Push and open PR to main AND develop git push origin hotfix/fix-order-minimum-validation # 5. Deploy hotfix branch to staging and verify manually # 6. Get code review approval (at least 1 reviewer) # 7. Merge to main → triggers CI/CD to production # 8. Merge to develop → keeps develop branch in sync

Common Semester 4 Bug Categories

Bug TypeExampleFix Approach
Business logic bugDelivery fee not applied to COD ordersFix service layer + add unit test for COD path
UI bugCart total overflows on small screensCSS fix + add responsive test case
Data bugRestaurant rating shows NaN if no reviewsAdd null-safe calculation in rating service
Security patchexpress releases a security update (CVE)npm update express + full regression test
Performance bugAdmin analytics page times out at 30 sAdd DB index + cache the aggregation query
The "Minimal Fix" Principle

In production, make the smallest possible change that fixes the bug. Don't refactor surrounding code. Don't add unrelated features. The larger the change, the higher the risk of introducing new bugs.

2. E2E Test Automation (Playwright)

⭐ Good to Have

Automate full user journeys through the real browser

⭐ Good to Have — Adds significant value to your project report and impresses evaluators. Not required for a passing grade.

End-to-End (E2E) tests drive a real browser through complete user journeys — from login to order placement — and verify that the whole stack works together. They catch bugs that unit and integration tests miss.

What E2E tests cover that unit tests cannot

ScenarioUnit Test Catches?E2E Catches?
Cart total updates live as quantity changes❌ No (UI logic)✅ Yes
JWT token is stored after login❌ No (browser storage)✅ Yes
Order page is inaccessible when logged out❌ No (routing)✅ Yes
Payment button shows Razorpay modal❌ No (3rd party UI)✅ Yes
createOrder() returns correct data✅ Yes✅ Yes

Setup: Install Playwright

cd food-delivery-frontend npm install --save-dev @playwright/test npx playwright install chromium # install browser binaries # playwright.config.js is auto-created # Test files go in: tests/e2e/

E2E Test 1 — Customer Login Flow

// tests/e2e/auth.spec.js import { test, expect } from '@playwright/test'; test.describe('Customer Login', () => { test('successful login redirects to home', async ({ page }) => { await page.goto('/login'); await page.fill('[data-testid="email"]', 'customer@test.com'); await page.fill('[data-testid="password"]', 'password123'); await page.click('[data-testid="login-btn"]'); // Should redirect to home page await expect(page).toHaveURL('/'); await expect(page.locator('[data-testid="user-menu"]')).toBeVisible(); }); test('wrong password shows error message', async ({ page }) => { await page.goto('/login'); await page.fill('[data-testid="email"]', 'customer@test.com'); await page.fill('[data-testid="password"]', 'wrongpassword'); await page.click('[data-testid="login-btn"]'); await expect(page.locator('[data-testid="error-msg"]')) .toContainText('Invalid email or password'); await expect(page).toHaveURL('/login'); // stays on login page }); test('protected route redirects unauthenticated user', async ({ page }) => { // No login — go directly to orders await page.goto('/orders'); await expect(page).toHaveURL('/login'); }); });

E2E Test 2 — Place an Order (Happy Path)

// tests/e2e/order.spec.js import { test, expect } from '@playwright/test'; // Re-use a logged-in state so we don't repeat login steps test.use({ storageState: 'tests/e2e/auth-state.json' }); test('customer can add item and place order', async ({ page }) => { // 1. Browse to a restaurant await page.goto('/restaurants'); await page.click('[data-testid="restaurant-card"]:first-child'); // 2. Add one item to cart await page.click('[data-testid="add-to-cart"]:first-child'); await expect(page.locator('[data-testid="cart-count"]')).toHaveText('1'); // 3. Go to checkout await page.click('[data-testid="cart-icon"]'); await page.click('[data-testid="checkout-btn"]'); await expect(page).toHaveURL('/checkout'); // 4. Confirm order await page.click('[data-testid="place-order-btn"]'); // 5. Should land on order confirmation page await expect(page).toHaveURL(/\/orders\/\d+/); await expect(page.locator('[data-testid="order-status"]')) .toContainText('Confirmed'); });

Running E2E Tests

# Run all E2E tests (headed — you see the browser) npx playwright test --headed # Run headless (for CI) npx playwright test # Run a specific file npx playwright test tests/e2e/auth.spec.js # Show HTML report after run npx playwright show-report

Adding E2E to GitHub Actions CI

# .github/workflows/e2e.yml name: E2E Tests on: [push] jobs: e2e: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: { node-version: '20' } - name: Install dependencies run: npm ci - name: Install Playwright browsers run: npx playwright install --with-deps chromium - name: Start dev server run: npm run dev & - name: Run E2E tests run: npx playwright test - name: Upload Playwright report if: always() uses: actions/upload-artifact@v4 with: name: playwright-report path: playwright-report/

3. Performance Testing (k6)

⭐ Good to Have

Measure how your API behaves under load before users find out

⭐ Good to Have — Demonstrates production awareness. Even a basic load test with a screenshot in your report is impressive for MCA.

Performance testing checks how your system behaves under concurrent load — multiple users hitting the API simultaneously. It finds bottlenecks before real users do.

Performance Testing Terms

TermWhat it meansExample target
ThroughputRequests handled per second (RPS)> 50 RPS for student project
p95 response time95% of requests complete within this time< 500 ms
Error rate% of requests that return 5xx< 1%
Concurrent usersUsers hitting the system at the same time20–50 for student project

Install k6 (Load Testing Tool)

# macOS brew install k6 # Or run via Docker (no install needed) docker run --rm -i grafana/k6 run - <script.js

Load Test Script — Restaurant Listing API

// tests/perf/restaurant-list.js import http from 'k6/http'; import { check, sleep } from 'k6'; export const options = { stages: [ { duration: '30s', target: 10 }, // ramp up to 10 users over 30s { duration: '1m', target: 20 }, // stay at 20 users for 1 minute { duration: '20s', target: 0 }, // ramp down to 0 ], thresholds: { http_req_duration: ['p(95)<500'], // 95% of requests under 500ms http_req_failed: ['rate<0.01'], // less than 1% errors }, }; export default function () { const BASE_URL = 'https://your-app.render.com'; const res = http.get(`${BASE_URL}/api/restaurants?city=Mumbai`); check(res, { 'status is 200': (r) => r.status === 200, 'has restaurants array': (r) => JSON.parse(r.body).data.length > 0, 'response time < 500ms': (r) => r.timings.duration < 500, }); sleep(1); // simulate user think time }

Load Test Script — Full Order Flow

// tests/perf/order-placement.js import http from 'k6/http'; import { check, sleep } from 'k6'; export const options = { vus: 10, // 10 virtual users duration: '1m', // run for 1 minute }; const BASE_URL = 'https://your-app.render.com'; export function setup() { // Login once and share the token across VUs const res = http.post(`${BASE_URL}/api/auth/login`, JSON.stringify({ email: 'load-test@test.com', password: 'testpassword' }), { headers: { 'Content-Type': 'application/json' } }); return { token: JSON.parse(res.body).token }; } export default function (data) { const headers = { 'Content-Type': 'application/json', 'Authorization': `Bearer ${data.token}`, }; // Step 1: Browse restaurants const restaurants = http.get(`${BASE_URL}/api/restaurants`, { headers }); check(restaurants, { 'restaurants loaded': (r) => r.status === 200 }); sleep(0.5); // Step 2: Get menu const menu = http.get(`${BASE_URL}/api/restaurants/1/menu`, { headers }); check(menu, { 'menu loaded': (r) => r.status === 200 }); sleep(0.5); // Step 3: Place order const order = http.post(`${BASE_URL}/api/orders`, JSON.stringify({ restaurant_id: 1, items: [{ menu_item_id: 1, quantity: 2 }], delivery_address_id: 1 }), { headers }); check(order, { 'order placed successfully': (r) => r.status === 201, 'order has id': (r) => JSON.parse(r.body).data?.id != null, }); sleep(1); }

4. Code Hardening

⭐ Good to Have

Small, high-impact improvements before final submission

⭐ Good to Have — Takes 1–2 hours but significantly improves code quality and evaluator impression.

Input Validation — express-validator

Ensure all incoming data is validated at the API boundary before it reaches business logic or the database.

// src/validators/order.validator.js const { body, validationResult } = require('express-validator'); const validatePlaceOrder = [ body('restaurant_id').isInt({ min: 1 }).withMessage('Invalid restaurant'), body('items').isArray({ min: 1 }).withMessage('Cart cannot be empty'), body('items.*.menu_item_id').isInt({ min: 1 }), body('items.*.quantity').isInt({ min: 1, max: 20 }), body('delivery_address_id').isInt({ min: 1 }), (req, res, next) => { const errors = validationResult(req); if (!errors.isEmpty()) { return res.status(422).json({ errors: errors.array() }); } next(); } ]; module.exports = { validatePlaceOrder }; // Usage in route: // router.post('/orders', authenticate, validatePlaceOrder, orderController.create);

Rate Limiting — express-rate-limit

// src/middleware/rateLimiter.js const rateLimit = require('express-rate-limit'); // Strict limit for auth endpoints (prevent brute force) const authLimiter = rateLimit({ windowMs: 15 * 60 * 1000, // 15 minutes max: 10, // max 10 attempts per IP message: { error: 'TOO_MANY_ATTEMPTS', message: 'Try again in 15 minutes' }, }); // General API limit const apiLimiter = rateLimit({ windowMs: 60 * 1000, // 1 minute window max: 100, // 100 requests per minute per IP }); module.exports = { authLimiter, apiLimiter }; // In app.js: // app.use('/api/auth', authLimiter); // app.use('/api', apiLimiter);

Graceful Shutdown

Handle SIGTERM properly so in-flight requests finish before the server stops. This is important for zero-downtime deployments.

// src/server.js const app = require('./app'); const { PrismaClient } = require('@prisma/client'); const prisma = new PrismaClient(); const server = app.listen(process.env.PORT || 3000, () => { console.log(`Server running on port ${process.env.PORT || 3000}`); }); // Graceful shutdown process.on('SIGTERM', async () => { console.log('SIGTERM received — closing server gracefully'); server.close(async () => { await prisma.$disconnect(); console.log('Database connection closed'); process.exit(0); }); // Force exit after 10 seconds if still not closed setTimeout(() => process.exit(1), 10_000); });
