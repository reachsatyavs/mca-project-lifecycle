1. WBS / Task Breakdown

Decomposing the project into manageable work packages

A Work Breakdown Structure (WBS) decomposes the total project scope into smaller, manageable deliverables. Each "leaf" node is a work package — small enough to estimate and assign to one person.

WBS for Online Food Delivery System

Level 1Level 2Level 3 (Work Packages)
1. Planning & Requirements1.1 IdeationProject proposal, feasibility report, team charter
1.2 SRSFunctional reqs, NFRs, user stories, use cases, business rules
1.3 SPMPWBS, timeline, risk register, role assignments
1.4 Design DocsArchitecture diagram, ER diagram, class diagrams, tech stack decision
1.5 SetupGitHub repo, project board (Jira/Trello), coding conventions doc
2. Backend Development2.1 Auth ModuleRegistration, login, JWT/session, password reset
2.2 Restaurant ModuleCRUD restaurant, CRUD menu items
2.3 Order ModuleCart, place order, order status FSM, order history
2.4 Payment ModulePayment gateway integration, invoice generation
2.5 Admin ModuleApprove restaurants, dashboard analytics
3. Frontend Development3.1 Customer PortalHome, restaurant listing, menu, cart, checkout, tracking
3.2 Restaurant DashboardLogin, order management, menu management
3.3 Delivery Partner AppOrder view, status updates
3.4 Admin PanelUser management, restaurant approval, reports
4. Testing4.1 Unit TestingTest all API endpoints (PyTest / JUnit)
4.2 Integration TestingOrder flow end-to-end, payment flow
4.3 UATUser acceptance testing with 5 external testers
5. Deployment5.1 CI/CDGitHub Actions pipeline, auto-deploy to staging
5.2 Production DeployDeploy to Render/Railway, domain setup, SSL
5.3 MonitoringSet up Sentry, UptimeRobot, structured logging
6. Documentation6.1 Technical DocsAPI docs (Swagger), TDD, database schema doc
6.2 User DocsUser manual, deployment guide, final project report
WBS Rule of Thumb

Each leaf-level work package should be completable by one person in 1–5 days. If a task takes longer than 5 days, break it down further. This makes estimation and progress tracking much easier.

2. SPMP Document Structure

What goes into a Software Project Management Plan

The SPMP is a living document — it is created in Semester 1 and updated each semester. It is the single source of truth for how the project is managed.

Standard SPMP Sections

SectionContents for Online Food Delivery
1. Project OverviewProject name, objective, one-paragraph description, key stakeholders (team, faculty guide)
2. Project DeliverablesList all outputs per semester: SRS, TDD, working software, test reports, deployment
3. WBSHierarchical task list (see above)
4. Schedule (Gantt Chart)Timeline showing which tasks happen when, with dependencies
5. Resource PlanWho is on the team, their roles, hours per week committed
6. Risk Management PlanRisk register (see below)
7. Quality PlanCode review process, test coverage targets, definition of "done"
8. Communication PlanMeeting cadence, reporting format, escalation path
9. Configuration ManagementGit branching strategy, version numbering, change request process
10. Monitoring & ControlHow progress is tracked (Jira/Trello board), velocity measurement

Gantt Chart Tip

Create a Gantt chart in ProjectLibre (free) or Google Sheets. Show each work package as a bar spanning its planned start-to-end dates. Mark milestones as diamonds. Update it monthly.

4. Risk Register

Identifying, assessing, and planning responses for project risks

A Risk Register lists everything that could go wrong and defines how to respond. Each risk has a probability (High/Medium/Low) and impact (High/Medium/Low). Priority = Probability × Impact.

Risk IDRisk DescriptionProbImpactPriorityMitigation StrategyOwner
R-01Team member drops course or becomes unavailableMediumHighHighCross-train all members; document code clearly; inform faculty earlyPM
R-02Scope creep — stakeholders keep requesting new featuresHighMediumHighEnforce Non-Goals document; use formal Change Request processPM
R-03Payment gateway API changes or sandbox discontinuedLowHighMediumAbstract payment logic behind an interface; have a mock payment fallbackBackend Dev
R-04Free hosting tier has downtime during demoMediumHighHighMonitor uptime; have a recorded demo video as backup; test before presentationDevOps
R-05Data loss due to no backup strategyLowHighMediumDaily automated DB backups; store backups on Google DriveDB Designer
R-06Security vulnerability in authenticationMediumHighHighUse proven libraries (e.g., bcrypt, JWT); run OWASP ZAP scan in Sem 4Backend Dev
R-07Frontend–backend API integration delaysHighMediumHighDefine API contracts (OpenAPI spec) early in Sem 2; use mock APIs during frontend devPM
R-08Performance too slow for demo load testMediumMediumMediumAdd DB indexes early; use caching (Redis); performance test in Sem 3Backend Dev
