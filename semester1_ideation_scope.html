1. Project Selection

Choosing the right project — and articulating the problem it solves

What is an Online Food Delivery System?

An Online Food Delivery System is a web-based platform that connects customers who want to order food with restaurants that prepare the food and delivery partners who bring it to the customer's doorstep. Think of it as a digital marketplace layered between the restaurant and the hungry customer.

Real-world examples your students are familiar with: Swiggy, Zomato, DoorDash, and Uber Eats. Your project will be a simplified, educational version of the same concept — but every layer from database to UI will be built by the team.

The Problem Statement

Problem Statement

Customers in a city/campus face inconvenience in discovering nearby restaurants, viewing their menus, and ordering food efficiently. Restaurants lack a digital channel to reach a wider audience. The absence of a unified, technology-driven platform means missed business opportunities for restaurants and frustration for customers.

Why This Project for MCA Students?

This project is an excellent choice because it:

  • Covers all major software engineering concepts: authentication, CRUD, payments, real-time status, APIs.
  • Has well-known real-world counterparts students can reference for inspiration.
  • Is complex enough to last 4 semesters yet scoped to be achievable by a student team.
  • Involves multiple user roles (Customer, Restaurant Owner, Delivery Partner, Admin) — excellent for learning role-based access control.
  • Produces a portfolio-worthy project that is directly relevant to industry.

System Overview (Student Scope)

ActorWhat they do in the system
CustomerRegisters, browses restaurants & menus, adds to cart, places orders, tracks delivery, pays, leaves reviews.
Restaurant OwnerRegisters restaurant, manages menu (add/edit/delete items), sets prices, views & accepts incoming orders.
Delivery PartnerRegisters as partner, views assigned orders, updates pickup & delivery status.
Platform AdminApproves restaurants, manages users, views analytics, handles complaints.
Student Tip

Start by sketching a simple diagram with these four actors and draw arrows showing what each one does. This becomes the foundation of your Use Case diagram in the Requirements phase.

2. Goals & Non-Goals

Explicitly stating what the project will AND will not do

A Goals & Non-Goals document prevents scope creep — the tendency of a project to grow beyond its original plan. Writing down what is out of scope is just as important as writing what is in scope.

Goals (In Scope)

Customer-Facing

  • User registration & login (email + password)
  • Browse restaurants by cuisine or location
  • View restaurant menus with item photos & prices
  • Add/remove items from a cart
  • Place orders and choose delivery address
  • Simulated online payment (sandbox gateway)
  • Real-time order status tracking (Pending → Accepted → Preparing → Out for Delivery → Delivered)
  • View order history
  • Rate and review restaurants

Restaurant & Admin

  • Restaurant owner registration & profile management
  • Menu management (CRUD for food items)
  • Order notification and acceptance/rejection
  • Delivery partner registration
  • Order assignment to delivery partner
  • Admin panel: approve/reject restaurants, view reports
  • Email notifications (order confirmation, status updates)
  • Responsive web application (desktop + mobile browser)

Non-Goals (Explicitly Out of Scope)

Out of Scope for Student Project
  • No native mobile apps (iOS / Android) — web-responsive design is sufficient.
  • No real payment gateway — use a sandbox (Razorpay test mode / Stripe test mode).
  • No live GPS tracking — status updates are manual; real-time GPS is too complex.
  • No multi-city support — scope to one city or campus area.
  • No AI/ML recommendations — no "you might also like" engine.
  • No multi-language support — English only for the prototype.
  • No third-party logistics integration (FedEx, Dunzo etc.).
  • No franchise / chain restaurant management — single-outlet restaurants only.
  • No subscription or loyalty rewards programme.
Why Non-Goals Matter

When a stakeholder says "Can we also add a chatbot?", you can point to the Non-Goals document and say it is out of scope. Without this list, your project will never be finished. A Change Request (CR) process in Semester 2 handles any genuine changes that arise later.

3. Feasibility Check

Assessing whether the project can realistically be built

A Feasibility Study answers one question: "Can we actually build this?" It examines the project from three angles — technical, operational, and economic.

Technical Feasibility

ConcernAssessment for Online Food DeliveryVerdict
Programming language & framework Python (Express.js/Flask), Java (Spring Boot), or Node.js (Express) — all well-documented with large communities. ✅ Feasible
Database MySQL or PostgreSQL for structured data (orders, users, restaurants). Both free and widely taught. ✅ Feasible
Frontend HTML/CSS/JS with React or plain Bootstrap — no proprietary tool required. ✅ Feasible
Payment integration Razorpay or Stripe sandbox offers free test environments with full API docs. ✅ Feasible
Hosting / Deployment Render, Railway, or Heroku free tier — no cost for student projects. ✅ Feasible
Real-time features Order status polling (refresh every 10 s) is simpler than WebSockets and fully feasible for student scope. ✅ Feasible

Operational Feasibility

  • Team size: 4–5 MCA students covering backend, frontend, DB, QA, and DevOps roles.
  • Duration: 4 semesters (~18–24 months) — adequate to cover all phases.
  • Supervision: Faculty guide provides feedback at end of each semester milestone.
  • Tools access: VS Code, Git/GitHub, Postman — all free and available in college labs.
  • Knowledge prerequisite: Databases (taught in Sem 1), Web programming (Sem 1), DBMS lab — all prerequisite coursework aligns.

4. Constraints List

Known limitations that the project must work within

Constraints are hard limits that cannot be negotiated. They differ from risks (which are uncertain) and non-goals (which are choices). Documenting constraints early prevents unrealistic expectations later.

Time Constraints

SemesterDurationKey Deliverable
Semester 1~6 monthsSRS, SPMP, system design documents, architecture diagram
Semester 2~6 monthsWorking backend + database; basic frontend prototype
Semester 3~6 monthsFull integrated working system; alpha testing complete
Semester 4~6 monthsProduction-deployed, tested, documented final product

Technology Constraints

  • Must use technologies covered in the MCA curriculum (no proprietary/licensed tools without approval).
  • No real money transactions — all payment must go through sandbox/test mode.
  • System must be deployable to a free cloud hosting service.
  • Source code must be version-controlled on GitHub/GitLab (public or private — as instructed).

Team Constraints

  • Team size: minimum 3, maximum 5 students.
  • Each team member must contribute to documentation, not just coding.
  • A project guide (faculty) must be assigned and approve major phase deliverables.

Quality Constraints

  • Each semester must produce a documented deliverable (SRS, TDD, Test Report, etc.).
  • Code must follow a consistent naming convention and style guide chosen in Semester 1.
  • Minimum 70% test coverage for core modules in Semester 3.
  • Final system must be accessible via a public URL (not just localhost).
Constraint vs. Risk

A constraint is certain — e.g., "we have 6 months." A risk is uncertain — e.g., "a team member might drop the course." Both need to be documented, but constraints go in the SPMP while risks go in the Risk Register (covered in Planning).

5. Team Formation

Defining roles, responsibilities, and how the team will work together

A well-structured team with clear role boundaries reduces confusion and improves accountability. For a 4–5 member MCA team, the following roles are recommended. A single person can wear two hats, but each responsibility must have one clear owner.

Recommended Roles

RoleResponsibilitiesKey Skills Needed
Project Manager / Scrum Master Maintains task tracker (Jira/Trello), runs sprint meetings, writes SPMP, communicates with faculty guide, tracks milestones. Organization, communication, documentation
Backend Developer Builds REST APIs, implements business logic (order placement, payment flow), manages database interactions, handles authentication. Python/Java/Node.js, SQL, REST API design
Frontend Developer Builds the web UI (customer portal, restaurant dashboard, admin panel), integrates with backend APIs, ensures responsive design. HTML, CSS, JavaScript (React or Vanilla JS)
Database Designer / Full-Stack Designs ER diagram, creates schema, writes migrations, optimizes queries, also supports backend or frontend as needed. SQL, ER modeling, query optimization
QA & DevOps Writes test plans and test cases, performs manual and automated testing, sets up CI/CD pipeline (Semester 3), manages deployments. Testing tools (Selenium/PyTest), GitHub Actions, Docker basics

Team Charter

A Team Charter is a short agreement the team writes in Semester 1. It covers working norms and prevents future conflict.

Sample Team Charter Items
  • Weekly sync meeting: every Monday, 6 PM, 30 minutes (online / in lab).
  • All code commits go through a Pull Request with at least one reviewer.
  • Decisions are made by team consensus; PM has tie-breaking vote.
  • Any team member missing 2 consecutive meetings informs the faculty guide.
  • Communication channel: WhatsApp group for quick updates, GitHub Issues for task tracking.
  • Code style guide: ESLint (Python) / Google Java Style (Java) — agreed on Day 1.
