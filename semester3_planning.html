Go-Live Checklist

Everything that must be verified before going to production

A Go-Live Checklist prevents the embarrassment of launching a broken system. Every item must be checked off before the production deployment in Semester 4.

Functional Checks

  • ☐ Customer can register, login, and update profile.
  • ☐ Customer can browse restaurants, view menus, and add items to cart.
  • ☐ Customer can place an order (online payment + COD paths both work).
  • ☐ Restaurant owner can accept orders and manage menu.
  • ☐ Delivery partner can mark orders as Picked Up and Delivered.
  • ☐ Customer receives email on order confirmation and status changes.
  • ☐ Admin can approve/reject restaurants.
  • ☐ Customer can rate and review after delivery.

Non-Functional Checks

  • ☐ All pages load in < 3 seconds on 4G (tested with Chrome DevTools throttling).
  • ☐ System handles 50 concurrent users (basic load test with Locust).
  • ☐ HTTPS is enforced — all HTTP redirects to HTTPS.
  • ☐ Passwords are hashed with bcrypt (confirmed via DB inspection).
  • ☐ No sensitive data in frontend JS (no API keys in browser code).
  • ☐ Mobile responsive: tested on 375px, 768px, 1280px breakpoints.

Operational Checks

  • ☐ Sentry is configured and receiving error events.
  • ☐ UptimeRobot is monitoring the production URL.
  • ☐ Database backup is running daily and verified by restoring once.
  • ☐ All environment variables are in production — no hardcoded values.
  • ☐ CI/CD pipeline is green on the main branch.
  • ☐ Rollback procedure is documented and tested.
The Cost of Skipping the Checklist

Going live without a checklist is how teams discover on demo day that the payment gateway is still in sandbox mode, or that the admin password is "admin123", or that the database is pointing to localhost. The checklist takes 30 minutes to run and saves 3 hours of embarrassment.
