Semester 1 — Foundation

Prototype & Proof of Concept

Example Project: Online Food Delivery System

In Semester 1, coding is limited to building a small prototype or Proof of Concept (PoC) to validate key technical decisions — not to build the full system.

🧪

Why Build a Prototype First?

Reduce technical uncertainty before committing to full development

A Proof of Concept (PoC) is a small piece of code that proves a critical technical idea works. It is throwaway code — not production quality. Its purpose is to answer one or two high-risk technical questions before full development begins in Semester 2.

PoC vs. Prototype vs. MVP
  • PoC: "Can this technology work for us at all?" — small, focused, throwaway.
  • Prototype: "What will this look/feel like?" — slightly bigger, may include UI mockups. Still throwaway.
  • MVP (Minimum Viable Product): "Can users use a minimal version?" — production-quality, but minimal features. This is your Semester 2/3 goal.

Recommended Semester 1 PoC Goals

For the Online Food Delivery System, your Semester 1 PoC should answer these questions:

Sample PoC File Structure

food_delivery_poc/ ├── backend/ │ ├── app.py # Flask/Express.js app entry point │ ├── models.py # Restaurant model │ ├── routes/ │ │ └── restaurants.py # GET /api/restaurants │ └── package.json ├── frontend/ │ └── index.html # Fetch API call to backend └── README.md # "How to run this PoC"
Key Rule

Keep the PoC simple. Two or three API endpoints and a single HTML page is enough. The goal is confidence in your tech choices, not a feature-complete system. Full development starts in Semester 2.
