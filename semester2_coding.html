1. Core Modules

Breaking the backend into independently developable modules

The backend is structured as separate Express.js routers, each owning a slice of the domain. This separation makes development parallel — Member A can build the Order module while Member B builds the Auth module without conflicts.

2. Backend APIs

Building REST APIs with proper authentication and error handling

Good API design includes authentication guards, input validation, meaningful error responses, and consistent response format.

Sample: Order Placement API (Node.js / Express)

// src/controllers/order.controller.js const { validationResult } = require('express-validator'); const OrderService = require('../services/order.service'); const placeOrder = async (req, res, next) => { const errors = validationResult(req); if (!errors.isEmpty()) { return res.status(400).json({ error: 'VALIDATION_ERROR', fields: errors.array() }); } try { const order = await OrderService.placeOrder({ customerId: req.user.id, // set by auth middleware ...req.body }); return res.status(201).json({ order_id: order.id, status: order.status, total: order.total }); } catch (err) { next(err); // forwarded to global error handler } }; module.exports = { placeOrder };
// src/middleware/auth.middleware.js const jwt = require('jsonwebtoken'); const authenticate = (req, res, next) => { const header = req.headers.authorization; if (!header?.startsWith('Bearer ')) { return res.status(401).json({ error: 'AUTHENTICATION_REQUIRED' }); } try { const payload = jwt.verify(header.split(' ')[1], process.env.JWT_SECRET); req.user = payload; // { id, email, role } next(); } catch (err) { const code = err.name === 'TokenExpiredError' ? 'TOKEN_EXPIRED' : 'INVALID_TOKEN'; return res.status(401).json({ error: code }); } }; // Usage: router.post('/orders', authenticate, authorize('CUSTOMER'), placeOrder) const authorize = (...roles) => (req, res, next) => { if (!roles.includes(req.user.role)) { return res.status(403).json({ error: 'PERMISSION_DENIED' }); } next(); }; module.exports = { authenticate, authorize };

3. Database Layer

Implementing the schema, migrations, and data access layer

Use your ORM's migration system to manage schema changes. Never alter the database directly in production — always use migrations so changes are reproducible and version-controlled.

Prisma Migration Workflow (Node.js ORM)

# 1. Edit schema.prisma (add/change a model field) # 2. Generate and apply a migration npx prisma migrate dev --name add_order_status_field # 3. Prisma creates a migration file in prisma/migrations/ # 4. Commit schema.prisma AND the migration file to Git together # 5. On production deploy: npx prisma migrate deploy
// Sample prisma/schema.prisma (excerpt) model Order { id Int @id @default(autoincrement()) customerId Int restaurantId Int status OrderStatus @default(PLACED) total Decimal @db.Decimal(10, 2) paymentMode PaymentMode createdAt DateTime @default(now()) items OrderItem[] payment Payment? customer User @relation(fields: [customerId], references: [id]) } enum OrderStatus { PLACED ACCEPTED PREPARING OUT_FOR_DELIVERY DELIVERED CANCELLED }

Transactions for Order Placement

Order placement involves multiple DB writes (create order, create order items, update payment). These must be wrapped in a database transaction — if any step fails, the whole operation is rolled back.

// src/services/order.service.js const { prisma } = require('../config/database'); async function placeOrder({ customerId, restaurantId, items, paymentMode, addressId }) { const total = items.reduce((sum, i) => sum + i.price * i.quantity, 0); // prisma.$transaction guarantees all-or-nothing const order = await prisma.$transaction(async (tx) => { const newOrder = await tx.order.create({ data: { customerId, restaurantId, paymentMode, addressId, total, items: { create: items } } }); await tx.payment.create({ data: { orderId: newOrder.id, amount: total, status: 'PENDING', mode: paymentMode } }); return newOrder; }); return order; // Only committed if no exception was thrown inside }

5. Code Reviews

Learning from each other and catching bugs before they reach develop

Every Pull Request (PR) must be reviewed by at least one other team member before merging. Code review is not about criticism — it's about knowledge sharing and catching bugs early.

Code Review Checklist

  • Does the code follow the agreed style guide (ESLint / Prettier config)?
  • Are all inputs validated and sanitised?
  • Are there any hardcoded secrets or credentials?
  • Are error cases handled with appropriate status codes?
  • Are database queries optimised (no N+1 queries)?
  • Are unit tests included or updated?
  • Is the code understandable without excessive comments?
  • Does the PR description link to the Jira/GitHub issue?
N+1 Query Problem

If you load 10 orders and then loop through them to fetch each restaurant separately, that's 1 + 10 = 11 database queries. With Prisma, use include: { restaurant: true, items: true } to fetch everything in 1–2 queries via a JOIN. This is one of the most common performance bugs in beginner code.
