1. Detailed Use Cases

Expanding high-level use cases into full interaction flows

In Semester 1 you listed the use cases. In Semester 2 you expand the most complex ones into step-by-step interaction sequences — these guide backend and frontend developers on exactly what to build.

UC-12: Delivery Partner — Accept and Deliver Order

ActorDelivery Partner
TriggerSystem assigns order to partner
PreconditionPartner is online and within 3 km of restaurant; order status is ACCEPTED by restaurant
Main Flow 1. System assigns order → partner sees notification.
2. Partner views order details (restaurant address, item list, customer delivery address).
3. Partner navigates to restaurant and picks up order.
4. Partner taps "Picked Up" → order status becomes PICKED_UP.
5. Partner navigates to customer address and delivers order.
6. Partner taps "Delivered" → order status becomes DELIVERED.
7. System sends delivery confirmation to customer.
Alternate — Partner RejectsPartner rejects within 3 minutes → system re-assigns to next nearest available partner.
PostconditionOrder is DELIVERED; customer prompted to rate restaurant; partner's delivery count incremented.

2. API Contract Draft

Defining the REST API interface before writing a single line of backend code

An API Contract is the agreement between the backend and frontend teams. By defining the API shape (endpoint, method, request body, response) before coding, both teams can work in parallel — the frontend uses mock data matching the contract while the backend builds the real implementation.

🔴 Standard Error Envelope

All error responses follow a consistent structure so the frontend can handle them uniformly:

{ "error": "ERROR_CODE_IN_CAPS", // Machine-readable code for frontend logic "message": "Human-readable message", // Shown directly to user "status_code": 400, // Same as HTTP status code "details": { ... } // Optional: extra context (field errors, limits, etc.) }
HTTP StatusWhen to UseExample in Food Delivery
200 OKSuccessful GET / PATCHRestaurant list returned; order status updated
201 CreatedSuccessful POST that creates a resourceOrder placed; review submitted; user registered
400 Bad RequestInvalid input from clientCart below ₹100; missing required field; invalid transition
401 UnauthorizedMissing or invalid tokenNo JWT sent; JWT expired
403 ForbiddenValid token but wrong role/permissionCustomer tries to call restaurant-only endpoint
404 Not FoundResource doesn't existOrder ID doesn't exist; restaurant not found
409 ConflictDuplicate / business rule conflictUser already reviewed this order; email already registered
500 Server ErrorUnhandled backend exceptionDatabase connection lost; uncaught exception — never expose stack trace!
Tool: OpenAPI / Swagger

Write your API contract as an OpenAPI 3.0 YAML file. Use Swagger Editor (editor.swagger.io) to view it visually. With Express.js (Node.js), install swagger-ui-express and the spec is auto-generated from your code. You can also paste these JSON examples directly into Swagger as example responses.

3. Acceptance Criteria

Specific, testable conditions that define "done" for each feature

Acceptance Criteria (AC) are precise statements that a feature must satisfy to be considered complete. They bridge the gap between user stories and test cases. They are written by the PM/BA in Semester 2 before the developer starts coding each feature.

Acceptance Criteria: Order Placement Feature

Feature: Customer Places an Order
  • AC-1: Customer cannot place an order if cart is empty — "Place Order" button is disabled.
  • AC-2: Customer cannot place an order if restaurant is currently closed — error shown.
  • AC-3: Customer must select a delivery address before proceeding to payment.
  • AC-4: Payment page displays the correct itemised breakdown (subtotal, delivery fee, taxes, total).
  • AC-5: After successful online payment, order status becomes "PLACED" and order ID is shown.
  • AC-6: After failed payment, order is NOT created in the database.
  • AC-7: Order confirmation email is sent within 2 minutes of placement.
  • AC-8: Restaurant receives an in-app notification within 5 seconds of order being placed.
  • AC-9: Cart is cleared after successful order placement.
  • AC-10: Customer is redirected to the order tracking page after placement.

These AC items directly become your Semester 2 test cases — one test per AC item.
