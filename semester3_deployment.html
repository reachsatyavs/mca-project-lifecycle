1. CI Pipeline — GitHub Actions

Running tests automatically on every Pull Request

A Continuous Integration (CI) pipeline automatically runs your tests every time code is pushed to GitHub. If tests fail, the PR is blocked from merging — preventing broken code from reaching the main branch.

GitHub Actions Workflow

# .github/workflows/ci.yml name: CI — Test & Lint on: push: branches: [develop, main] pull_request: branches: [develop] jobs: test: runs-on: ubuntu-latest services: postgres: image: postgres:15 env: POSTGRES_DB: fooddelivery_test POSTGRES_USER: testuser POSTGRES_PASSWORD: testpass ports: ['5432:5432'] options: --health-cmd pg_isready --health-interval 10s steps: - uses: actions/checkout@v4 - name: Set up Python 3.11 uses: actions/setup-python@v5 with: python-version: '3.11' - name: Install dependencies run: | pip install -r package.json pip install jest-cov - name: Run linter (flake8) run: flake8 apps/ --max-line-length=100 - name: Run tests with coverage env: DATABASE_URL: postgresql://testuser:testpass@localhost:5432/fooddelivery_test SECRET_KEY: test-secret-key DJANGO_SETTINGS_MODULE: config.settings.testing run: | jest --cov=apps --cov-fail-under=70 --cov-report=xml - name: Upload coverage report uses: codecov/codecov-action@v4

This workflow runs lint + all tests on every PR. A red ❌ on the CI check tells the reviewer not to merge. A green ✅ means the code is safe to review.

3. Docker Containers

Packaging the application for consistent deployment

Docker packages your application and all its dependencies into a container — a lightweight, isolated environment that runs identically on any machine or cloud server.

Dockerfile (Express.js Backend)

FROM python:3.11-slim WORKDIR /app # Install system dependencies RUN apt-get update && apt-get install -y \ libpq-dev gcc && \ rm -rf /var/lib/apt/lists/* # Install Python dependencies COPY package.json . RUN pip install --no-cache-dir -r package.json # Copy application code COPY . . # Collect static files RUN python manage.py collectstatic --noinput # Run with pm2 (production WSGI server) CMD ["pm2", "config.wsgi:application", \ "--bind", "0.0.0.0:8000", \ "--workers", "2", \ "--timeout", "60"]
Container Best Practices
  • Add .dockerignore to exclude .env, node_modules/, __pycache__/.
  • Never bake secrets into the Docker image — pass them as environment variables at runtime.
  • Use a non-root user inside the container for security.
  • Keep the image small: use python:3.11-slim not python:3.11.

4. Automated Deploys

Auto-deploying to staging on every merge to develop

With Continuous Deployment (CD), every merge to the develop branch automatically deploys to staging — no manual steps needed. This keeps staging always up-to-date.

Render Auto-Deploy Configuration

Render can watch your GitHub repository and auto-deploy when a branch is updated:

  • Go to Render dashboard → Select your service → Settings.
  • Set "Auto-Deploy" to ON, branch = develop.
  • Every push to develop triggers: pull latest code → run migrations → restart service.
  • Deployment takes ~2–3 minutes. Render shows a deployment log.

CD Flow (Develop Branch)

Developer pushes to feature branch │ ▼ GitHub Actions CI runs (lint + tests) │ PASS FAIL │ │ ▼ ▼ PR reviewed PR blocked (tests must pass first) │ ▼ Merge to develop │ ▼ Render auto-deploy to staging │ ▼ QA verifies on staging.food-delivery.render.com │ ▼ (when stable) Manual deploy to production (Semester 4)
