Scope Refinements

Adjustments discovered once actual development begins

After 6 months of planning (Semester 1), the Semester 2 team reviews the original scope with fresh eyes — now they know which features are harder than expected and which non-goals should be reconsidered.

Common Refinements for Food Delivery System

Original Scope ItemRefinementReason
Real-time order tracking via auto-refreshChanged from polling (refresh every 15s) to WebSocket for status updatesPolling caused unnecessary server load; WebSocket is cleaner with Socket.io
Image upload to local storageChanged to Cloudinary (cloud-based image hosting)Local storage doesn't persist on Render's free tier (ephemeral filesystem)
Single address per userExtended to multiple addresses with a "default" flagUser testing showed this was a major usability complaint
Admin approval: manual email processAdmin approval via in-app dashboard with notificationEmail-only approach was too slow; in-app is faster and trackable

Change Request (CR) Process

The formal way to change scope without causing chaos

A Change Request (CR) is a formal document submitted whenever the project scope, design, or plan needs to change. Without a CR process, changes are made informally, leading to undocumented features, broken designs, and missed deadlines.

CR Template

CR IDCR-001
TitleAdd real-time order status via WebSocket
Requested ByBackend Developer (Member B)
DateWeek 3, Semester 2
DescriptionReplace polling-based order status with WebSocket (Socket.io) for real-time push updates to the customer's browser when order status changes.
Impact on ScopeAdds a new technical component (Socket.io). Affects FR-304 (Order Status Tracking). No new user-facing features.
Impact on Timeline+3 days for Member B to integrate Socket.io
Impact on NFRsImproves NFR-2 (API response time) — status updates now push instead of pull
Decision✅ Approved by PM and Faculty Guide on [date]
SRS Update Required?Yes — FR-304 updated to reflect WebSocket approach
CR Process Steps
  1. Team member identifies a change and fills in a CR form.
  2. PM reviews the CR, estimates impact on time and scope.
  3. Faculty guide approves or rejects the CR.
  4. If approved: update relevant SRS/TDD sections and create tasks on the board.
  5. Log the CR in a Change Log (a running table of all CRs).
