PRD Final

Product Requirements Document — the definitive feature specification

The PRD (Product Requirements Document) is an updated, final version of the SRS from Semester 1 — incorporating all approved Change Requests and the more detailed acceptance criteria written in Semester 2. While the SRS uses IEEE format, the PRD uses a product-manager style format focused on user value.

PRD Structure for Food Delivery

SectionContents
Executive SummaryWhat the product is, who it's for, why we're building it (1 page)
User PersonasDetailed profiles of Customer, Restaurant Owner, Delivery Partner, Admin with goals and pain points
Feature ListAll features with MoSCoW priority (Must/Should/Could/Won't) and completion status
User Stories & Acceptance CriteriaUpdated stories from Semester 1 + new stories discovered in Semester 2
Non-Functional RequirementsUpdated NFRs with measurable targets confirmed against actual system
Change LogAll CRs applied since Semester 1 SRS with rationale

TDD Final

Technical Design Document — updated to reflect actual implementation

The TDD v2.0 updates the Semester 1 design documents to reflect the actual system as built — including any design decisions made during coding, updated ER diagrams, and finalised sequence diagrams.

  • Architecture diagram updated to show all actual components (including Sentry, Cloudinary, SendGrid).
  • Final ER diagram exported from actual database schema (use pg_dump or dbdiagram.io import).
  • API design section with link to live Swagger documentation URL.
  • Sequence diagrams for all key flows (order placement, payment, delivery).
  • Technology decisions documented with version numbers (Express.js 4.2.7, PostgreSQL 15.2, etc.).

API Docs (Swagger)

Auto-generated, live API documentation from code

Swagger (OpenAPI) documentation is generated automatically from your API code using swagger-ui-express. It provides an interactive UI at /api/schema/swagger-ui/ where developers can explore and test all endpoints.

Setup with Express.js (Node.js)

pip install swagger-ui-express # settings.py INSTALLED_APPS += ['drf_spectacular'] REST_FRAMEWORK = { 'DEFAULT_SCHEMA_CLASS': 'drf_spectacular.openapi.AutoSchema', } SPECTACULAR_SETTINGS = { 'TITLE': 'Online Food Delivery API', 'DESCRIPTION': 'REST API for the Online Food Delivery System', 'VERSION': '2.0.0', } # urls.py from drf_spectacular.views import SpectacularAPIView, SpectacularSwaggerView urlpatterns += [ path('api/schema/', SpectacularAPIView.as_view(), name='schema'), path('api/schema/swagger-ui/', SpectacularSwaggerView.as_view(), name='swagger-ui'), ]

Add docstrings to your views for richer documentation:

from drf_spectacular.utils import extend_schema class OrderCreateView(APIView): @extend_schema( request=OrderCreateSerializer, responses={201: OrderResponseSerializer}, description="Place a new food order. Customer must be authenticated.", tags=["Orders"] ) def post(self, request): ...
Deploy Swagger to Staging

Keep Swagger UI accessible on your staging URL (https://your-app.render.com/api/schema/swagger-ui/) so your faculty guide can see all your APIs without setting up Postman. Disable it in production (or add Basic Auth) to avoid exposing your API to the public.
