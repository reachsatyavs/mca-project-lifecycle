Testing — Unit, Integration & Test Plan

Example Project: Online Food Delivery System

Testing starts in Semester 2 alongside coding — not after. Unit tests verify individual functions; integration tests verify that modules work together. The test plan documents the overall testing strategy.

1. Unit Tests

Testing individual functions and classes in isolation

A unit test tests a single function or method in isolation — no database, no network, no file system. External dependencies are replaced with mocks. Unit tests run in milliseconds and should be run before every commit.

What to Unit Test in Food Delivery

  • Order total calculation (subtotal + delivery fee + tax)
  • Business rule: minimum order ₹100 validation
  • Business rule: free delivery threshold ≥ ₹500
  • Order status transition rules (only valid transitions allowed)
  • Password validation (length, complexity)
  • Distance calculation between two lat/lng coordinates
  • Delivery partner assignment logic (nearest partner)

Example Unit Test (Jest — Node.js)

// tests/unit/order.service.test.js const { getDeliveryFee, calculateOrderTotal } = require('../../src/services/order.service'); const { validateMinimumOrder } = require('../../src/utils/validators'); describe('Order Calculation', () => { test('delivery fee is ₹40 for orders below ₹500', () => { expect(getDeliveryFee(450)).toBe(40); }); test('delivery is free at exactly ₹500', () => { expect(getDeliveryFee(500)).toBe(0); }); test('delivery is free above ₹500', () => { expect(getDeliveryFee(650)).toBe(0); }); test('order total includes delivery fee and tax', () => { const total = calculateOrderTotal({ subtotal: 300, taxRate: 0.05 }); expect(total).toBe(300 + 40 + 15); // subtotal + fee + tax }); test('throws if cart total is below ₹100 minimum', () => { expect(() => validateMinimumOrder(80)).toThrow('Minimum order amount is ₹100'); }); }); describe('Order Status Transitions', () => { test('PLACED can transition to ACCEPTED', () => { expect(canTransitionTo('PLACED', 'ACCEPTED')).toBe(true); }); test('DELIVERED cannot go back to PREPARING', () => { expect(canTransitionTo('DELIVERED', 'PREPARING')).toBe(false); }); });
Coverage Target

Aim for 70–80% code coverage on the business logic layer (services, validators). Run npm test -- --coverage to generate a coverage report. Jest will show you which branches and lines are not yet tested.

2. Integration Tests

Testing that modules work correctly together, including the database

Integration tests test the interaction between components — typically a full HTTP request through the API layer down to the test database. They are slower than unit tests but catch bugs that unit tests miss (e.g., database constraint violations, serialisation issues).

Example Integration Test: Full Order Flow (Jest + Supertest)

// tests/integration/order.test.js const request = require('supertest'); const app = require('../../src/app'); const { prisma } = require('../../src/config/database'); describe('POST /api/orders/', () => { let customerToken, restaurantId, menuItemId, addressId; beforeAll(async () => { // Seed test data directly via Prisma const customer = await prisma.user.create({ data: { name: 'Test Customer', email: 'test@customer.com', passwordHash: '', role: 'CUSTOMER', emailVerified: true } }); const restaurant = await prisma.restaurant.create({ data: { name: 'Test Biryani House', status: 'APPROVED', ownerId: 1 } }); const item = await prisma.menuItem.create({ data: { name: 'Chicken Biryani', price: 250, restaurantId: restaurant.id, isAvailable: true } }); const address = await prisma.address.create({ data: { userId: customer.id, street: 'MG Road', city: 'Bangalore' } }); restaurantId = restaurant.id; menuItemId = item.id; addressId = address.id; customerToken = generateTestJWT(customer); // helper that signs a real JWT }); afterAll(() => prisma.$disconnect()); test('places order successfully with COD', async () => { const res = await request(app) .post('/api/orders/') .set('Authorization', `Bearer ${customerToken}`) .send({ restaurantId, addressId, paymentMethod: 'COD', items: [{ menuItemId, quantity: 2 }] }); expect(res.status).toBe(201); expect(res.body.status).toBe('PLACED'); const order = await prisma.order.findFirst(); expect(order).not.toBeNull(); }); test('returns 400 when items array is empty', async () => { const res = await request(app) .post('/api/orders/') .set('Authorization', `Bearer ${customerToken}`) .send({ restaurantId, addressId, paymentMethod: 'COD', items: [] }); expect(res.status).toBe(400); expect(res.body.error).toBe('VALIDATION_ERROR'); }); });

4. Write Test Cases

Semester 2 is when formal test cases are written — not executed yet

Writing vs Executing — Key Distinction

In Semester 2 you write the test cases — formally, from the requirements. Each TC gets a TC ID, precondition, test data, steps, and expected result. The Status field stays "Not Executed" until Semester 3 when the full system is deployed and all TCs are run against it. Think of it like preparing exam questions now, and the exam being held next semester.

Write test cases as soon as a feature's API endpoints are merged to develop. Writing them immediately after coding ensures your test data and expectations are accurate while the feature is fresh.
