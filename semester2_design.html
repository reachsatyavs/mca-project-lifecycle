1. Flow Diagrams

Mapping the business process flows of key operations

🤔 Students Often Confuse These Three — Here is the Difference

All three live in the Technical Design Document (TDD), but each answers a completely different question:

Diagram Question it answers Shapes used Lives in
ER Diagram What data do we store?
Entities, attributes, relationships in the database		 Entity boxes with columns; crow's foot lines (1:M, 1:1) Database design / TDD
Flow Diagram What steps happen, in what order?
Business process logic, decisions, branches		 Ovals (start/end), rectangles (steps), diamonds (decisions), arrows Process design / TDD
Class Diagram What code objects exist and how do they relate?
Classes, attributes, methods, inheritance		 3-part rectangles; hollow arrows (inheritance); lines (association) OOP design / TDD
Sequence Diagram Which components talk to each other, in what order?
System interactions over time (Browser ↔ API ↔ DB)		 Vertical lifelines; horizontal arrows (calls and responses) Interaction design / TDD

Rule of thumb: Draw an ER diagram for every database you design. Draw a flow diagram for every complex business process (order placement, refund, partner assignment). Draw a class diagram for your key domain objects. Draw a sequence diagram for every API call that touches 3+ components.

📖 What is a Flow Diagram (Flowchart)?

A Flow Diagram (Flowchart) is a visual representation of a process using standardised shapes connected by arrows. It shows the sequence of steps, decision points, and possible paths through a workflow.

ShapeMeaningExample in Food Delivery
Stadium / Pill ([text])Start / End (Terminator)Customer opens checkout; Order accepted
Rectangle [text]Process / Action stepAdd item to cart; Send confirmation email
Diamond {text}Decision (condition)Cart total ≥ ₹100? Restaurant open?
Rounded rect (text)Sub-process / NoteRazorpay popup; Auto-retry logic
Arrow --> with labelFlow + condition labelYes / No on decision branches

2. Sequence Diagrams

Showing interactions between system components over time

📖 What is a Sequence Diagram?

A Sequence Diagram (UML) shows which components talk to each other, in what order, and what messages they exchange — across time (top = earliest, bottom = latest).

ElementMeaning
Lifeline (vertical dashed line)Represents a participant — e.g., Browser, Backend API, Database, Razorpay
Activation bar (rectangle on lifeline)Shows when a component is actively processing
Solid arrow →A call / request being sent
Dashed arrow ⤍A response / return value
Alt / Opt / Loop framesConditional paths, optional blocks, and repeated steps
X on lifelineComponent terminates or an error kills the flow

When to draw: Draw sequence diagrams for every multi-component interaction: user login, order placement, payment, delivery status update. Crucially, always draw both the success path and the error path.

Solid arrows are calls/requests; dashed arrows are responses. Activation bars show when a component is actively processing.

Sequence Diagram 2: Payment Failure — Error Path ❌

An error path diagram is just as important as the happy path. Notice that no DB write happens — the order is never created when payment fails.

%%{init: {'theme': 'dark'}}%% sequenceDiagram actor Customer participant FE as Frontend participant API as Backend API participant DB as Database participant RZP as Razorpay Customer->>+FE: Click Checkout FE->>+API: POST /api/orders/ API->>+DB: Validate cart DB-->>-API: ✅ Valid API->>+RZP: Create payment order RZP-->>-API: {gateway_order_id} API-->>-FE: 200 {gateway_order_id} FE-->>Customer: Show Razorpay popup Note over Customer,RZP: ❌ Card declined by bank Customer->>FE: Enter card details FE->>+API: POST /api/payments/verify/ {error: BAD_REQUEST} Note over API,DB: ⛔ No DB write — order NOT created API-->>-FE: 400 {error: PAYMENT_FAILED, retry_allowed: true} FE-->>Customer: Show error + Retry / Pay with UPI
Key Rule: Never Create an Order Before Payment Succeeds

The order row is only inserted after the payment signature is verified as genuine. This prevents "ghost orders" — orders in the DB that were never paid for. Verify first, persist second.

Sequence Diagram 3: Expired JWT — Authentication Error ❌

Every protected endpoint must handle an expired token gracefully. The frontend silently refreshes the token and retries — the user never sees the error.

3. API Design (OpenAPI / Swagger)

Finalising and documenting your REST API contract

In Semester 2, the API contract draft from requirements becomes a formal OpenAPI 3.0 specification. This is both documentation and a mock server source for frontend development.

REST API Design Principles for Food Delivery

PrincipleExample
Use nouns, not verbs in URLs✅ GET /orders/   ❌ GET /getOrders/
Use plural nouns✅ /restaurants/   ❌ /restaurant/
Nest resources sensibly✅ GET /restaurants/7/menu/
Use HTTP verbs correctlyGET=read, POST=create, PUT/PATCH=update, DELETE=remove
Return appropriate status codes200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 404 Not Found, 500 Server Error
Version your API/api/v1/orders/ — so future breaking changes go to /api/v2/

4. DB Schema (Final)

The finalised, indexed, production-ready database schema

The ER diagram from Semester 1 becomes a final SQL schema in Semester 2. At this stage you also add indexes for performance and foreign key constraints for data integrity.

Critical Indexes to Add

TableIndex Column(s)Why
usersemail (UNIQUE)Login lookup; uniqueness constraint
restaurantsstatus, cuisine_typeFiltering approved restaurants by cuisine
menu_itemsrestaurant_id, is_availableFast menu lookup for a restaurant
orderscustomer_id, status, created_atCustomer order history; status filtering
ordersrestaurant_id, statusRestaurant's pending orders view
order_itemsorder_idFetching line items for an order
reviewsrestaurant_idCalculating restaurant average rating
Over-indexing Warning

Don't index every column. Each index speeds up reads but slows down writes (INSERT/UPDATE). Index columns that appear in WHERE clauses, JOIN conditions, or ORDER BY in your most frequent queries.

5. Wireframes / UI Design

Sketching the user interface before writing frontend code

Wireframes are low-fidelity sketches of each page — they show layout and navigation without worrying about colours or fonts. Create wireframes for every major screen before the frontend developer starts coding.

Key Screens to Wireframe

  • Customer: Home/Search, Restaurant Listing, Restaurant Menu, Cart, Checkout, Order Tracking, Order History, Profile.
  • Restaurant Owner: Dashboard, Incoming Orders, Menu Management (add/edit/delete item).
  • Delivery Partner: Available Orders, Active Order (with status update buttons).
  • Admin: Pending Restaurant Approvals, All Users list, Platform Analytics.
Wireframing Tools

Figma (free for students) is the industry standard. Excalidraw (free, browser-based) is great for quick hand-drawn style sketches. Even paper wireframes scanned as images in your TDD are acceptable at student level.
