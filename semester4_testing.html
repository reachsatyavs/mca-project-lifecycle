Semester 4 — Production

Testing — UAT, Regression Suite & Security Scan

Example Project: Online Food Delivery System

Semester 4 testing focuses on final validation before production: User Acceptance Testing by stakeholders, a full regression suite to verify nothing is broken, and a security scan to identify vulnerabilities.

👤

UAT — User Acceptance Testing

Stakeholders validate the system meets their expectations

UAT is the final testing phase where the "business" (in your case: faculty guide + project sponsor) confirms the system satisfies the original requirements. It is not a developer activity — developers fix issues found here, but don't run the tests.

UAT Plan for Food Delivery

Register and place first orderFaculty guide (as customer)Order confirmed within 30 seconds; confirmation email received
Restaurant owner accepts orderFaculty guide (as restaurant)Order visible on dashboard; accepted within 5 minutes; customer status updates
Full payment flow (online payment)Faculty guideRazorpay sandbox payment succeeds; invoice emailed

System performance under 20 usersQA team (load test)All pages load under 3 seconds

UAT Sign-Off

At the end of UAT, the faculty guide signs a UAT Acceptance Certificate — a single-page document confirming all UAT scenarios passed. This is a critical project deliverable for the final submission.

🔁

Regression Suite

Ensuring new code hasn't broken existing functionality

A regression test suite is a set of tests that verify all previously working functionality still works after new code changes. In Semester 4, every hotfix and patch must pass the full regression suite before being deployed to production.

Regression Suite for Food Delivery

  • All 89 unit tests (run automatically via CI/CD).
  • All 35 integration tests (run automatically via CI/CD).
  • Critical E2E scenarios (run manually before each production deploy):
    • Customer end-to-end order flow (add to cart → pay → track).
    • Restaurant order acceptance flow.
    • Delivery partner pickup and deliver flow.
    • Admin restaurant approval flow.
    • Password reset flow.
Automated Regression in CI

Unit and integration tests run automatically on every PR (GitHub Actions). E2E tests can be run in GitHub Actions on a schedule (e.g., every night) using Playwright's headless browser mode. This gives you continuous regression coverage without manual effort.

Finding and fixing vulnerabilities before production launch

Outdated DependenciesNo known CVEs in dependenciespip audit / npm audit

Run OWASP ZAP (Free)

# Install OWASP ZAP (Docker) docker pull ghcr.io/zaproxy/zaproxy:stable # Run baseline scan against your staging URL docker run -t ghcr.io/zaproxy/zaproxy:stable zap-baseline.py \ -t https://food-staging.render.com \ -r zap_report.html

ZAP will scan your staging site and produce an HTML report listing all found vulnerabilities by severity. Fix High and Medium severity issues before production launch.
