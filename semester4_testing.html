Security Testing — "Good to Have" for MCA Students
UAT and Regression are required deliverables for every MCA project. Security testing is above and beyond — most student projects skip it entirely. If you include even 5–6 security test cases with OWASP references and evidence of fixes, it will significantly impress evaluators and differentiates your project from others. The good news: the basic tests are easy to run once you know what to look for.

1. UAT — User Acceptance Testing

Stakeholders validate the system meets their expectations

UAT is the final testing phase where the "business" (faculty guide + project sponsor) confirms the system satisfies the original requirements. It is not a developer activity — developers fix issues found here, but don't run the tests.

UAT Plan for Food Delivery

ScenarioTesterAcceptance Condition
Register and place first orderFaculty guide (as customer)Order confirmed within 30 seconds; confirmation email received
Restaurant owner accepts orderFaculty guide (as restaurant)Order visible on dashboard; accepted; customer status updates
Full online payment flowFaculty guideRazorpay sandbox succeeds; invoice emailed
Admin approves a restaurantFaculty guide (as admin)Restaurant visible to customers after approval
System performance under 20 usersQA team (load test)All pages load under 3 seconds
UAT Sign-Off Certificate

At the end of UAT, the faculty guide signs a UAT Acceptance Certificate — a one-page document confirming all scenarios passed and the system is accepted for production. This is a critical deliverable for the final submission report.

2. Regression Suite

Ensuring new code hasn't broken existing functionality

A regression suite re-verifies all previously working functionality after new code changes. In Semester 4, every hotfix must pass the full regression suite before deploying to production.

Regression Suite for Food Delivery

  • All unit tests (Jest) — run automatically on every PR via GitHub Actions.
  • All integration tests (Jest + Supertest) — run automatically via CI.
  • Critical E2E scenarios run manually before each production deploy:
    • Customer end-to-end: add to cart → pay → track → delivered
    • Restaurant owner: accept order → mark ready
    • Delivery partner: pickup → deliver → status update
    • Admin: approve restaurant, suspend user
    • Password reset flow
Automated Regression in CI/CD

Unit and integration tests run on every PR (GitHub Actions). Schedule Playwright E2E tests to run nightly: schedule: cron: '0 2 * * *' in your workflow YAML. This gives continuous regression coverage without manual effort.

3. Security Testing Overview

⭐ Good to Have

What to test, the OWASP Top 10, and how to approach it

⭐ Good to Have — Sections 3, 4, and 5 are all optional but will strongly impress evaluators if included.

Security testing verifies that the system is resistant to attacks. For a food delivery app, the two most valuable things to protect are user data (PII, payment info) and business logic (orders can only be placed by verified customers, not bots).

4. Security Test Cases

Concrete test cases for the most impactful vulnerabilities

How to Read These Test Cases

Each test case below is something a malicious user or attacker might try. Your job is to confirm that the system correctly blocks or rejects the attack. A PASS here means the attack was stopped. A FAIL means a real vulnerability exists that must be fixed.

Category 1 — Broken Access Control (IDOR)

SEC-01 · OWASP A01 IDOR
Customer A reads Customer B's order details HIGH
ScenarioEvery order has an integer ID (e.g., /api/orders/42). If the API doesn't check ownership, Customer A can read any order by guessing the ID.
PreconditionTwo accounts exist: customer_a@test.com (owns order 42), customer_b@test.com (owns order 43).
Steps1. Login as Customer B → get JWT token.
2. GET /api/orders/42 with Customer B's token.
3. Inspect response.
ExpectedHTTP 403 · error: "PERMISSION_DENIED" · no order data returned.
Actual (buggy)HTTP 200 · Full order details of Customer A's order returned. ← This is a real vulnerability.
FixIn the order query: WHERE id = :id AND customer_id = req.user.id — always scope queries to the authenticated user.
Verified byRe-run SEC-01 after fix → must return 403.
SEC-02 · OWASP A01 Broken Access Control
Customer calls an Admin-only endpoint HIGH
ScenarioThe admin endpoint PATCH /api/admin/restaurants/:id/approve should only be callable by users with role=ADMIN.
Steps1. Login as a regular customer → get token.
2. PATCH /api/admin/restaurants/7/approve with customer's token.
3. Inspect response.
ExpectedHTTP 403 · error: "PERMISSION_DENIED"
Actual (buggy)HTTP 200 · Restaurant approved. ← Authorization not checked.
FixEnsure authorize('ADMIN') middleware wraps every admin route. Add automated test that asserts CUSTOMER role receives 403 on all /api/admin/* routes.
SEC-03 · OWASP A01 Mass Assignment
User promotes themselves to Admin role via API body HIGH
ScenarioIf the update-profile endpoint blindly passes the request body to the ORM, an attacker can add "role": "ADMIN" to the update payload.
Steps1. Login as customer.
2. PATCH /api/users/me with body: {"name": "Priya", "role": "ADMIN"}.
3. GET /api/users/me and check the role field.
ExpectedRole remains "CUSTOMER". The role field is silently ignored (not a 400 error, just not updated).
FixUse an explicit allowlist of updatable fields: const { name, phone } = req.body — never spread the whole body into a DB update.

Category 2 — Authentication & JWT Vulnerabilities

SEC-04 · OWASP A07 JWT Tampering
Attacker tampers with JWT payload to change their role to Admin HIGH
ScenarioA JWT has three parts: header.payload.signature. If the server doesn't verify the signature properly, an attacker can decode the payload, change "role":"CUSTOMER" to "role":"ADMIN", and re-encode it.
Steps1. Login as customer → copy the access token.
2. Go to jwt.io → paste token → modify payload: "role": "ADMIN".
3. Change the signing algorithm to "none" (Algorithm: none attack).
4. Use the modified token to call GET /api/admin/users/.
5. Inspect response.
ExpectedHTTP 401 · error: "INVALID_TOKEN" — signature verification fails.
FixUse jwt.verify(token, process.env.JWT_SECRET, { algorithms: ['HS256'] }) — explicitly whitelist the algorithm. Never accept "alg: none".
SEC-05 · OWASP A07 Brute Force
Attacker brute-forces login with 100 password attempts MEDIUM
ScenarioIf there's no rate limiting on POST /api/auth/login, an attacker can try thousands of passwords per minute against any account.
Steps1. Write a loop sending 100 POST /api/auth/login requests in 10 seconds with different passwords.
2. Observe whether all 100 requests are processed or if the API starts rejecting them.
ExpectedAfter 5 failed attempts within 5 minutes, subsequent attempts return HTTP 429 Too Many Requests.
FixInstall express-rate-limit: limiter = rateLimit({ windowMs: 5*60*1000, max: 5, message: 'Too many login attempts' }) — apply to the login route only.
SEC-06 · OWASP A07 Token Expiry
Expired JWT is accepted after 24 hours MEDIUM
Steps1. Login and save the access token.
2. Wait until the token is past its expiry (or manually set a past exp in a test).
3. Make an API call with the expired token.
ExpectedHTTP 401 · error: "TOKEN_EXPIRED"
FixSet expiresIn: '15m' when signing access tokens. Implement a refresh token endpoint so the frontend can silently renew.

Category 3 — Injection

SEC-07 · OWASP A03 SQL Injection
SQL injection via restaurant search parameter MEDIUM
ScenarioIf the restaurant search is built by concatenating user input into a SQL string, an attacker can extract or destroy data.
Steps1. GET /api/restaurants?search=Biryani' OR '1'='1
2. Observe whether all restaurants are returned (injection worked) or just restaurants with "Biryani" in the name.
ExpectedReturns only restaurants matching "Biryani' OR '1'='1" as a literal string — which is zero results. No data leak.
FixUse Prisma's parameterised queries: prisma.restaurant.findMany({ where: { name: { contains: searchTerm } } }) — Prisma always uses parameterised queries. Never use raw template literals with user input.
SEC-08 · OWASP A03 XSS
Stored XSS via restaurant review text LOW
ScenarioIf review text is stored and displayed without sanitisation, an attacker can inject a script that runs in every other user's browser when they view the review.
Steps1. Submit a review with comment: <script>alert('XSS')</script>
2. Open the restaurant's review page in a browser.
3. Observe whether an alert box appears.
ExpectedThe text <script>…</script> is rendered as literal text, not executed. No alert box.
FixIn the frontend: use React's JSX (auto-escapes HTML) or textContent instead of innerHTML. In the backend: sanitise with the dompurify or validator package before saving.

Category 4 — Sensitive Data & Dependency Issues

SEC-09 · OWASP A02 Crypto Failure
Passwords stored in plain text in the database HIGH
Steps1. Register a new user with password TestPass@123.
2. Directly query the users table in the database.
3. Check the password_hash column for user.
ExpectedColumn shows a bcrypt hash like $2b$12$eImiTXuWVxfM37uY4JANj. — not the original password.
Fixconst hash = await bcrypt.hash(password, 12) — always hash with bcrypt (cost factor ≥ 10) before storing. Never store plain or MD5/SHA1 hashed passwords.
SEC-10 · OWASP A06 Known CVEs
npm packages contain known vulnerabilities LOW
Steps1. Run npm audit in the project root.
2. Note any High or Critical severity advisories.
Expected0 high/critical vulnerabilities. If any exist, npm audit fix resolves most automatically.
FixRun npm audit fix. For breaking changes: npm audit fix --force then re-run tests. Add npm audit --audit-level=high as a step in your GitHub Actions CI to block PRs with high CVEs.

5. OWASP ZAP Automated Scan

Running a free automated security scanner against your staging environment

OWASP ZAP (Zed Attack Proxy) is a free, open-source security scanner maintained by OWASP. Run it against your staging URL — never against production or other people's sites.

Run OWASP ZAP (Docker — no install needed)

# Pull the official OWASP ZAP image docker pull ghcr.io/zaproxy/zaproxy:stable # Run a baseline scan — passive checks, no active attacks # Replace the URL with your staging URL docker run -t ghcr.io/zaproxy/zaproxy:stable zap-baseline.py \ -t https://food-staging.render.com \ -r zap_report.html # Run a full active scan (more thorough — takes 10–30 min) docker run -t ghcr.io/zaproxy/zaproxy:stable zap-full-scan.py \ -t https://food-staging.render.com \ -r zap_full_report.html

Reading the ZAP Report

Alert LevelActionExample
HIGHMust fix before production launchSQL injection found, missing auth on endpoint
MEDIUMFix before final submissionMissing Content-Security-Policy header, no rate limiting
LOWNote in project report; fix if time permitsMissing X-Content-Type-Options header

Quick Security Headers Fix (Express.js)

// src/app.js — add before route registration const helmet = require('helmet'); app.use(helmet()); // helmet() sets 15 security headers in one line, including: // Content-Security-Policy // X-Content-Type-Options: nosniff // X-Frame-Options: DENY // Strict-Transport-Security (HSTS) // X-XSS-Protection // Rate limiting for auth endpoints const rateLimit = require('express-rate-limit'); const loginLimiter = rateLimit({ windowMs: 5 * 60 * 1000, // 5 minutes max: 5, // max 5 attempts per window message: { error: 'TOO_MANY_ATTEMPTS', retryAfter: '5 minutes' } }); app.use('/api/auth/login', loginLimiter);
What to Include in Your Project Report

To impress evaluators, include a 2-page "Security Testing" appendix in your final report: (1) OWASP ZAP scan results screenshot (before & after fixes), (2) table of security TCs with PASS/FAIL, (3) npm audit output showing 0 high vulnerabilities, (4) brief paragraph explaining how bcrypt, JWT expiry, and helmet protect the system. This takes one afternoon and sets your project apart from 90% of other MCA submissions.
