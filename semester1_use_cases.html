What is a Use Case?

Modelling system interactions between actors and the system

Use Case vs User Story
AspectUser StoryUse Case
Format"As a… I want… so that…"Structured table with actor, flows, conditions
Detail levelHigh-level, one sentenceStep-by-step, all alternate paths
AudienceProduct owner, stakeholdersDevelopers, testers, designers
PurposeWhat to build (the goal)How the system handles it (the interaction)

Actors in Online Food Delivery System

Customer Restaurant Owner Delivery Partner Platform Admin «system» Payment Gateway «system» Email Service

Actors are entities that interact with the system from outside. Primary actors initiate a use case (e.g., Customer places an order). Secondary actors are involved in supporting the use case but don't initiate it (e.g., Payment Gateway processes the payment).

Detailed Use Case: UC-08 Place Order

The most complex and critical use case in the system

Use Case IDUC-08
Use Case NamePlace Order
ActorCustomer (Primary), Payment Gateway (Secondary), Email Service (Secondary)
DescriptionCustomer reviews their cart, selects a delivery address, chooses a payment method, and confirms the order. The system processes the payment and creates an order record.
Preconditions1. Customer is logged in.  2. Cart contains at least one item totalling ≥ ₹100.  3. Customer has at least one saved delivery address.  4. Restaurant is currently open.
Main Flow 1. Customer clicks "Proceed to Checkout."
2. System displays order summary (items, quantities, subtotal, delivery fee, taxes, total).
3. Customer selects or adds a delivery address.
4. Customer selects payment method (Online / COD).
5. Customer clicks "Place Order."
6a. If Online payment: System creates Razorpay order → customer pays in popup → system verifies signature.
6b. If COD: Skip to step 7.
7. System creates order record with status "PLACED."
8. System sends order confirmation email to customer.
9. System notifies restaurant of new order.
10. System clears the cart and redirects customer to order tracking page.
Alternate Flow A Payment Failure (Step 6a): Gateway returns failure → System shows error with retry option → Order is NOT created → Cart is NOT cleared.
Alternate Flow B Restaurant Closed: System detects restaurant is outside operating hours → "Place Order" button is disabled → Message: "Spice Garden is currently closed. Opens at 10:00 AM."
Alternate Flow C Cart Below Minimum: System detects cart total < ₹100 → "Checkout" button shows tooltip: "Add ₹X more to proceed." → Button is disabled.
PostconditionsOrder record exists in DB with unique order ID. Cart is cleared. Restaurant receives a notification. Email sent to customer.
Related FRsFR-303, FR-401, FR-402, FR-403
Related User StoriesUS-C04 (Cart), US-C05 (Payment)

Detailed Use Case: UC-02 Login

Authentication — the gateway to all other use cases

Use Case IDUC-02
Use Case NameLogin
ActorAny registered user (Customer, Owner, Partner, Admin)
DescriptionA registered user provides their email and password. The system authenticates them and returns a JWT access token granting access to role-specific features.
Preconditions1. User has a registered account.  2. Email is verified.  3. Account is not suspended.
Main Flow 1. User navigates to the login page.
2. User enters email and password.
3. User clicks "Login."
4. System validates credentials (bcrypt hash comparison).
5. System issues a JWT access token (15 min) and a refresh token (7 days).
6. System updates user's last_login timestamp.
7. System redirects to role-specific dashboard (Customer → Home, Admin → Admin Panel).
Alternate Flow A Wrong Password: Returns error "Invalid email or password." No indication of which field is wrong (security). User can try again or use Forgot Password.
Alternate Flow B Account Suspended: System displays "Your account has been suspended. Contact support." No token issued.
Alternate Flow C Email Not Verified: System shows "Please verify your email. Resend verification link?" No token issued.
PostconditionsUser has valid JWT token. Session is active. last_login updated in DB.

Detailed Use Case: UC-12 Write Review

Post-delivery feedback loop — builds trust on the platform

Use Case IDUC-12
Use Case NameWrite Review
ActorCustomer (Primary)
DescriptionA customer who has received a delivered order submits a star rating and optional text review for the restaurant.
Preconditions1. Customer is logged in.  2. Order status = DELIVERED.  3. Customer has NOT already reviewed this specific order.
Main Flow 1. Customer opens order history and selects a delivered order.
2. System shows "Rate & Review" button (only for unreviewed delivered orders).
3. Customer selects a star rating (1–5).
4. Customer optionally enters text comment (max 500 characters).
5. Customer clicks "Submit Review."
6. System saves the review (customer_id, restaurant_id, order_id, rating, comment).
7. System recalculates restaurant average rating.
8. System shows confirmation: "Thank you for your review!"
Alternate Flow A Already Reviewed: "Rate & Review" button is hidden. API returns 409 Conflict if called directly.
Alternate Flow B Order Not Delivered: "Rate & Review" option not shown. API returns 400 with error REVIEW_NOT_ALLOWED.
PostconditionsReview record saved. Restaurant avg rating updated. Order marked as reviewed.
