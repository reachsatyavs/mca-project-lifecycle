1. Sprint Planning

Breaking Semester 2 into focused 2-week development sprints

A sprint is a fixed time period (typically 2 weeks) during which the team commits to completing a specific set of features. At the end of every sprint, there is a potentially shippable product increment.

Sprint 1 Definition of Done

  • All code reviewed via Pull Request and merged to main.
  • Unit tests pass (npm test) with ≥ 80% branch coverage (Jest).
  • No ESLint errors (npm run lint exits 0).
  • App deployed to staging URL and reachable via browser.
  • Health check endpoint returns { status: "ok" }.
  • Swagger docs updated for all new endpoints (/api-docs reachable on staging).
  • Sprint Review demo: team registers a new user, verifies email, logs in, and views JWT in response.

How Backlog Management Works

Story points measure effort/complexity, not hours. Use the Fibonacci scale: 1, 2, 3, 5, 8, 13. A 1-point task is trivial (e.g., update a text label). An 8-point task is complex (e.g., full payment integration).

During each Sprint Planning ceremony, the team pulls the top-priority backlog items into the upcoming sprint — only as many points as the team's velocity allows. Once Sprint 1 was done (13 points in 2 weeks), the team confirmed a velocity of ~13 points/sprint and pulled FOOD-004, 005, 006 into Sprint 2. The remaining items stay ordered in the backlog, ready for grooming before Sprint 3.

3. Revised Timeline

Updating the project timeline based on actual Sprint 1 velocity

After Sprint 1, you know your team's real velocity. If your planned 30 story points per sprint is actually 20, revise the timeline accordingly — do not pretend you are on track when you are not.

What Goes into the Revised Timeline

  • Updated Gantt chart with actual completion dates for completed work.
  • Recalculated remaining work based on actual velocity.
  • Any scope changes approved via CR reflected in the timeline.
  • Updated risk register: risks that materialised and new risks discovered.
Be Honest with Your Faculty Guide

If you are behind schedule, say so early. A revised timeline presented in Week 4 is manageable. A surprise reveal in Week 16 is a crisis. Good project management is about transparency, not perfection.

4. Change Request Log

Controlled tracking of every scope, requirement, or design change across all semesters

A Change Request (CR) is the formal mechanism for proposing any modification to the agreed baseline — whether that is a new feature, a removed feature, a revised requirement, or a design pivot. Every CR is logged, evaluated for impact, and either approved or rejected before any work begins. This prevents silent scope creep and keeps the SRS honest.

When to Raise a CR — Triggers
✅ A new requirement discovered during coding
✅ A feature that is technically infeasible as specified
✅ Faculty guide feedback requiring a design change
✅ A feature being descoped due to time pressure
✅ A bug fix that requires changing the SRS/design
✅ A third-party API or library change forcing a pivot
✅ Peer/beta feedback revealing a UX gap
✅ A security finding that changes how auth works
⚠️ CR vs Bug — Know the Difference
Change Request (CR)
The system does what the SRS says, but the SRS needs to change. Example: "The SRS says email-only login — we now want to add Google OAuth." That's a CR because the requirement itself is changing.
Bug / Defect
The system does NOT do what the SRS says. Example: "The SRS says login should return a JWT, but the API returns a session cookie instead." That's a bug — fix it, no CR needed.

Change Freeze & Escalation Policy

🔒 Change Freeze — End of Semester 3
No new features or scope additions after the Semester 3 code freeze date. Only bug fixes and performance improvements are permitted in Semester 4. Any CR raised after freeze is automatically deferred to the "future enhancements" section of the final report.
📣 Escalation Path
If the team cannot reach a unanimous decision on a CR within 2 working days, the Project Lead escalates to the Faculty Guide. Faculty Guide's decision is final. All escalations are logged in the CR form under "Decision" with the guide's rationale quoted directly.
Why Evaluators Value a CR Log

A Change Request Log demonstrates that your team operated with engineering discipline, not chaos. It shows: (1) you knew your baseline, (2) you evaluated trade-offs before changing it, (3) you got stakeholder sign-off, and (4) you kept documentation honest. Most MCA projects have zero change management — a well-maintained CR log immediately sets your project apart.

In the final viva, if an evaluator asks "Why does your system do X when the SRS says Y?" — your answer is: "CR-001, approved Week 3, here is the impact analysis and faculty sign-off." That is a professional answer.
