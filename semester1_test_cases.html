When Does Each Testing Activity Happen?

Understanding the testing lifecycle across semesters

Semester 1
📋 Test Planning
Define the testing strategy. What will be tested, how, by whom. Write the test plan document. Define scope, types of tests, environments, entry/exit criteria. Create the TC template.
You are here
Semester 2
✍️ Test Case Writing
Write formal test cases from the requirements. Each TC maps to a Use Case and Functional Requirement. Status at this stage: "Written — not yet executed." Unit tests and integration tests are also coded here.
Semester 3
▶️ Test Case Execution
Run all written test cases against the complete, integrated system. Record actual results — PASS or FAIL. File bugs for failures. Run regression after bug fixes. Produce the Test Summary Report.
Why This Order?

Test cases are written from requirements — so you need the requirements to be stable first (Semester 1). But you can't run them until the system is built (Semester 2 coding). And you want to run the full integrated system — not just individual modules — which is ready in Semester 3. This is the V-Model of software development.

The V-Model — Testing Mirrors Development

Left side (Build)PhaseRight side (Test)
Semester 1 — Requirements↔Semester 3 — Acceptance / System Testing
Semester 1 — Design↔Semester 3 — Integration Testing
Semester 2 — Coding↔Semester 2 — Unit & Integration Tests

Test Strategy

The high-level approach to quality assurance for the entire project

The test strategy is written in Semester 1 as part of the SRS / TDD and serves as the master agreement between the team on how quality will be achieved.

Testing Objectives

  • Verify that all 20 Use Cases from the SRS execute correctly as designed.
  • Verify that all 13 Functional Requirements are satisfied.
  • Verify that NFRs are met: page load < 2 s, API response < 500 ms, 99% uptime.
  • Ensure all business rules (BR-01 to BR-13) are enforced by the backend.
  • Detect and resolve all P1 and P2 defects before Semester 3 milestone review.

Test Scope & Types

What is in scope, what is out of scope, and what types of tests are planned

In Scope for Testing

✅ Functional (must test)
  • All authentication flows (register, login, JWT, refresh)
  • Restaurant CRUD and menu management
  • Order placement, payment, status transitions
  • Delivery partner assignment and delivery
  • Rating and review submission
  • Admin approval and user management
📊 Non-Functional (must measure)
  • API response time under normal load
  • Page load speed (Core Web Vitals)
  • Concurrent user handling (50 simultaneous)
  • Mobile responsiveness (375px → 1440px)
  • HTTPS enforcement
❌ Out of Scope
  • Razorpay / Stripe internal payment processing
  • SendGrid email delivery (third party)
  • Google Maps routing accuracy
  • Penetration testing (only OWASP ZAP in Sem 4)
  • Load testing beyond 100 concurrent users

Test Case Template

The standard format that every test case written in Semester 2 must follow

Important — This is a Template, Not an Actual Test Case

The placeholders below show the structure that every test case must follow when written in Semester 2. The Status field will have a value only after execution in Semester 3 — in Semester 2 it reads "Not Executed."

Test Case Template — Copy this for every TC you write in Semester 2
TC IDTC-<nn>  (e.g. TC-01, TC-02…)
TitleShort, specific description of what is being tested
CoversUC-XX · FR-YYY · US-ZZZ  ← traceability links
PreconditionDatabase state and test data that must exist before the test begins
Test DataExact values: email addresses, passwords, amounts, IDs
StepsNumbered, reproducible actions — step 1, step 2…
Expected ResultExact outcome: UI state, HTTP status code, DB row state, email sent
TypeHappy Path / Negative / Boundary / Security
PriorityP1 / P2 / P3
StatusNot Executed  (filled in Semester 3 as PASS / FAIL)
Bug IDBlank  (filled if status = FAIL, e.g. BUG-012)
