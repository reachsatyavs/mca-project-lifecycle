When Does Each Testing Activity Happen?

Understanding the testing lifecycle across semesters

Semester 1
📋 Test Planning
Define the testing strategy. What will be tested, how, by whom. Write the test plan document. Define scope, types of tests, environments, entry/exit criteria. Create the TC template.
You are here
Semester 2
✍️ Test Case Writing
Write formal test cases from the requirements. Each TC maps to a Use Case and Functional Requirement. Status at this stage: "Written — not yet executed." Unit tests and integration tests are also coded here.
Semester 3
▶️ Test Case Execution
Run all written test cases against the complete, integrated system. Record actual results — PASS or FAIL. File bugs for failures. Run regression after bug fixes. Produce the Test Summary Report.
Why This Order?

Test cases are written from requirements — so you need the requirements to be stable first (Semester 1). But you can't run them until the system is built (Semester 2 coding). And you want to run the full integrated system — not just individual modules — which is ready in Semester 3. This is the V-Model of software development.

The V-Model — Testing Mirrors Development

Left side (Build)PhaseRight side (Test)
Semester 1 — Requirements↔Semester 3 — Acceptance / System Testing
Semester 1 — Design↔Semester 3 — Integration Testing
Semester 2 — Coding↔Semester 2 — Unit & Integration Tests

Test Strategy

The high-level approach to quality assurance for the entire project

The test strategy is written in Semester 1 as part of the SRS / TDD and serves as the master agreement between the team on how quality will be achieved.

Testing Objectives

  • Verify that all 20 Use Cases from the SRS execute correctly as designed.
  • Verify that all 13 Functional Requirements are satisfied.
  • Verify that NFRs are met: page load < 2 s, API response < 500 ms, 99% uptime.
  • Ensure all business rules (BR-01 to BR-13) are enforced by the backend.
  • Detect and resolve all P1 and P2 defects before Semester 3 milestone review.

Test Scope & Types

What is in scope, what is out of scope, and what types of tests are planned

In Scope for Testing

✅ Functional (must test)
  • All authentication flows (register, login, JWT, refresh)
  • Restaurant CRUD and menu management
  • Order placement, payment, status transitions
  • Delivery partner assignment and delivery
  • Rating and review submission
  • Admin approval and user management
📊 Non-Functional (must measure)
  • API response time under normal load
  • Page load speed (Core Web Vitals)
  • Concurrent user handling (50 simultaneous)
  • Mobile responsiveness (375px → 1440px)
  • HTTPS enforcement
❌ Out of Scope
  • Razorpay / Stripe internal payment processing
  • SendGrid email delivery (third party)
  • Google Maps routing accuracy
  • Penetration testing (only OWASP ZAP in Sem 4)
  • Load testing beyond 100 concurrent users

Security Testing

Test cases that verify the system resists common security attacks — tied to NFR-4 through NFR-7

OWASP Top 10 Relevance

Security test cases are derived from the OWASP Top 10 — the industry standard list of the most critical web application security risks. For the Online Food Delivery System the highest-risk categories are: A01 Broken Access Control (role checks), A03 Injection (SQL injection), A07 Auth Failures (brute force, token abuse), and A03 XSS (stored cross-site scripting in reviews). Tool: OWASP ZAP for automated scanning; manual tests for IDOR and logic flaws.

Security Test Cases — Authentication & Access Control

SEC-01  OWASP A07 · Covers F-02 · UC-02 · NFR-4 · Sprint 1
SQL Injection on Login Endpoint
ThreatAttacker submits SQL payload in email/password to bypass authentication.
PreconditionLogin endpoint POST /api/auth/login is deployed.
Test Dataemail: ' OR '1'='1 · password: anything
Steps1. POST /api/auth/login with SQL payload.  2. Inspect HTTP response and DB logs.
Expected ResultHTTP 401 · error: "INVALID_CREDENTIALS" · No token issued · No DB error exposed in response · ORM parameterised query used (verify via query log).
How to PassUse Prisma / Sequelize ORM — never raw string SQL. Parameterised queries neutralise all injection.
PriorityP1
SEC-02  OWASP A03 · Covers F-07 · UC-12 · NFR-6 · Sprint 5
Stored XSS in Restaurant Review Comment
ThreatAttacker submits a script tag in a review comment; it executes in other users' browsers when they view the restaurant page.
PreconditionAuthenticated customer. Order ORD-001 status = DELIVERED.
Test Datacomment: <script>alert('XSS')</script> · rating: 3
Steps1. POST /api/reviews/ with XSS payload.  2. GET /api/restaurants/1/reviews/.  3. Render in browser and inspect.
Expected ResultScript tag is stored as escaped text (e.g., &lt;script&gt;) and is never executed in the browser. No alert box appears.
How to PassHTML-escape all user-supplied content before rendering. Use React's built-in escaping or DOMPurify for HTML output.
PriorityP1
SEC-03  OWASP A01 · Covers F-10, F-14, F-16 · NFR-7 · Sprint 2
Broken Access Control — Customer Calls Restaurant Management API
ThreatA customer (role = CUSTOMER) directly calls a restaurant owner endpoint to add a menu item or approve a restaurant, bypassing the UI role guards.
PreconditionCustomer JWT token for priya@example.com (role = CUSTOMER).
Test DataPOST /api/restaurants/5/menu-items/ with valid menu item payload. Bearer token = customer JWT.
Steps1. Obtain customer JWT.  2. Send POST /api/restaurants/5/menu-items/ with customer token.  3. Inspect response.
Expected ResultHTTP 403 Forbidden · error: "ACCESS_DENIED" · No menu item created in DB.
How to PassEvery API route must check req.user.role server-side. Middleware guard: requireRole('OWNER'). Never rely on UI hiding alone.
PriorityP1
SEC-04  OWASP A01 · Covers F-06, F-08 · NFR-7 · Sprint 4
IDOR — Customer Views Another Customer's Order
ThreatInsecure Direct Object Reference (IDOR): Customer A guesses Order ID and retrieves Customer B's order details, including address and payment method.
PreconditionCustomer A JWT for user A. Order ORD-050 belongs to Customer B (different user).
Test DataGET /api/orders/ORD-050/ with Customer A's JWT.
Steps1. Authenticate as Customer A.  2. GET /api/orders/ORD-050/ (Customer B's order).  3. Inspect response.
Expected ResultHTTP 403 Forbidden · error: "ACCESS_DENIED" · No order data returned for the wrong customer.
How to PassAlways filter DB queries by WHERE order.customer_id = req.user.id. Never trust the URL parameter alone.
PriorityP1
SEC-05  OWASP A07 · Covers F-02 · UC-02 · NFR-5 · Sprint 1
Brute Force Protection on Login
ThreatAttacker makes 100+ rapid login attempts with different passwords to guess a user's credentials.
PreconditionUser priya@example.com exists. Rate limiting middleware is deployed.
Test DataPOST /api/auth/login with wrong password repeated 15+ times within 60 seconds.
Steps1. Send 15 failed login attempts within 60 seconds.  2. Inspect response on the 16th attempt.
Expected ResultHTTP 429 Too Many Requests · error: "RATE_LIMIT_EXCEEDED" · Retry-After header present · Account NOT locked permanently (only throttled).
How to PassImplement rate limiting using express-rate-limit: max 15 attempts per 15 min per IP+email combination.
PriorityP2
SEC-06  OWASP A02 · NFR-5 · Sprint 1
HTTPS Enforcement — HTTP Redirects to HTTPS
ThreatData transmitted over plain HTTP can be intercepted (man-in-the-middle attack).
PreconditionApplication deployed on Render (staging) with custom domain. TLS certificate active.
Test DataURL: http://your-app.onrender.com/
Steps1. Open plain HTTP URL in browser (or use curl --max-redirs 0).  2. Observe redirect behaviour.
Expected ResultHTTP 301 or 308 redirect to HTTPS equivalent. No content served over plain HTTP. HSTS header present in HTTPS response: Strict-Transport-Security: max-age=31536000.
How to PassConfigure res.redirect middleware or set Render's "Force HTTPS" option. Add HSTS header in Express.
PriorityP2

Performance Test Cases

PERF-01  NFR-1 · Covers F-02 · UC-02 · Sprint 3
Home Page — Full Load Under 3 Seconds
GoalVerify that the customer-facing home page (restaurant listing) loads within 3 seconds on a standard connection.
Precondition50 approved restaurants in DB with menu items and images. Staging server running.
Test ToolGoogle Lighthouse (in Chrome DevTools) — Simulated Throttling: Fast 3G
Steps1. Open Chrome DevTools → Lighthouse tab.  2. Set throttling to "Fast 3G".  3. Run audit on /home.  4. Record LCP and Total Blocking Time.
Expected ResultLargest Contentful Paint (LCP) ≤ 3.0 s · Performance score ≥ 70 · No layout shift > 0.1
Pass / FailFAIL if LCP > 3 s. Fix: image compression, lazy loading, CDN for static assets.
PriorityP2
PERF-02  NFR-2 · Covers F-03 · UC-05 · Sprint 3
Restaurant Search API — Response Under 500 ms (Normal Load)
GoalVerify that the restaurant search endpoint responds within 500 ms under normal load of 10 virtual users.
Precondition50 restaurants in DB. Staging environment running. Database indexes on cuisine_type and status columns.
Test Toolk6 load testing script
k6 Scriptimport http from 'k6/http';
export const options = { vus: 10, duration: '30s' };
export default function () {
  const res = http.get('https://staging.api/restaurants/?cuisine=Indian');
  check(res, { 'status 200': r => r.status === 200, 'p95 < 500ms': r => r.timings.duration < 500 });
}
Expected Resultp95 response time ≤ 500 ms · p99 ≤ 800 ms · Error rate = 0% · 0 timeouts
Pass / FailFAIL if p95 > 500 ms. Fix: add DB index, implement Redis cache for restaurant listings.
PriorityP1
PERF-03  NFR-3 · Covers F-15 · UC-19 · Sprint 5
Concurrent Users — 50 Simultaneous Sessions Without Degradation
GoalVerify that the platform handles 50 simultaneous active users browsing and placing orders without significant latency increase.
PreconditionFull staging environment. 50 test user accounts seeded. Restaurants open. k6 installed.
Test Toolk6 — ramp-up scenario
k6 Scriptexport const options = {
  stages: [
    { duration: '1m', target: 50 }, // ramp up
    { duration: '3m', target: 50 }, // sustain
    { duration: '1m', target: 0 }, // ramp down
  ],
  thresholds: { http_req_duration: ['p95<500'] }
};
Expected Resultp95 latency ≤ 500 ms throughout load · Error rate ≤ 1% · No OOM or crash on server · CPU < 80% during sustained load
Pass / FailFAIL if error rate > 1% or p95 > 800 ms. Fix: horizontal scaling, DB connection pool sizing.
PriorityP2
PERF-04  NFR-2 · Covers F-05 · UC-08 · Sprint 3
Order Placement API — Response Under 500 ms Under Load
GoalVerify that the order placement API (the most DB-intensive endpoint) responds within 500 ms even when 20 users place orders simultaneously.
Precondition20 authenticated users (tokens pre-generated). Each has a valid cart. Restaurant open.
Test Toolk6 — POST /api/orders/ with authenticated payload
Steps1. Run k6 with 20 VUs each posting a valid order.  2. Record response times and DB transaction logs.  3. Check for deadlocks.
Expected ResultAll orders created successfully · HTTP 201 for each · p95 ≤ 500 ms · No database deadlock or constraint violation · No duplicate orders created
Pass / FailFAIL if any duplicate orders, deadlocks, or p95 > 800 ms. Fix: DB transactions with optimistic locking on inventory.
PriorityP1
