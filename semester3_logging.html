1. Minimum Viable Monitoring

The bare minimum every project must have before going live

You don't need Grafana and Prometheus for a student project. Three free tools give you 90% of what you need:

ToolWhat it gives youCostSetup time
SentryAutomatic error capture with stack traces; alerts on new errorsFree tier~15 min
UptimeRobotChecks your site every 5 minutes; emails you when it's downFree (50 monitors)~10 min
Render DashboardCPU, memory, and request graphs for your deployed serviceBuilt-in (free)Already there

Sentry Setup for Node.js / Express

npm install @sentry/node @sentry/profiling-node // src/instrument.js (import this FIRST in server.js) const Sentry = require('@sentry/node'); Sentry.init({ dsn: process.env.SENTRY_DSN, environment: process.env.NODE_ENV, // 'production' or 'development' tracesSampleRate: 0.2, // capture 20% of transactions for performance }); // src/app.js — add Sentry error handler AFTER all routes const Sentry = require('@sentry/node'); app.use(Sentry.Handlers.requestHandler()); // top of middleware app.use(Sentry.Handlers.errorHandler()); // bottom, before your own error handler

Health Check Endpoint (Required for UptimeRobot)

// src/routes/health.routes.js const { Router } = require('express'); const { PrismaClient } = require('@prisma/client'); const router = Router(); const prisma = new PrismaClient(); router.get('/health', async (req, res) => { let dbStatus = 'ok'; try { await prisma.$queryRaw`SELECT 1`; // lightweight DB ping } catch { dbStatus = 'error'; } const healthy = dbStatus === 'ok'; const httpStatus = healthy ? 200 : 503; res.status(httpStatus).json({ status: healthy ? 'healthy' : 'unhealthy', database: dbStatus, uptime: Math.floor(process.uptime()), // seconds server has been running version: process.env.npm_package_version || '1.0.0', timestamp: new Date().toISOString(), }); }); module.exports = router; // In app.js: app.use('/', healthRouter);

Point UptimeRobot to https://your-app.render.com/health — it checks every 5 minutes and emails you when the response is not 200 OK.

2. Health Dashboards

⭐ Good to Have

Visualising system health metrics in real time

⭐ Good to Have — A proper dashboard impresses evaluators and demonstrates production readiness. UptimeRobot + Render covers the basics for free; a custom metrics endpoint is the next level up.

Key Metrics to Dashboard

MetricWhy It MattersHealthy Range
HTTP 5xx error rateHigh rate = backend bugs or DB issues< 0.1% of requests
API p95 response timeSlow = DB queries need optimisation< 500 ms
Orders placed per hourBusiness health; also load indicatorBaseline after beta period
Active database connectionsToo many = connection pool exhausted< 80% of pool size
Memory usageMemory leak will grow over time< 70% of available RAM

Custom Metrics Endpoint (Extended Health)

// src/routes/health.routes.js — extended metrics endpoint router.get('/metrics', async (req, res) => { const [orderCount, activeUsers] = await Promise.all([ prisma.order.count({ where: { createdAt: { gte: new Date(Date.now() - 3_600_000) } } }), prisma.session.count({ where: { expiresAt: { gte: new Date() } } }), ]); const memUsage = process.memoryUsage(); res.json({ orders_last_hour: orderCount, active_sessions: activeUsers, memory_heap_mb: Math.round(memUsage.heapUsed / 1_048_576), uptime_seconds: Math.floor(process.uptime()), }); });

Free Dashboard Stack for Student Projects

  • UptimeRobot (free) — uptime monitoring + downtime alerts by email/SMS.
  • Render Dashboard — built-in CPU, memory, and request count for deployed services.
  • Sentry Performance — request duration histogram and slow transaction alerts.
  • Logtail (free 3 GB/month) — forward logs from Render, add a simple query UI.

3. Alert Rules

⭐ Good to Have

Getting notified when something goes wrong — before users report it

⭐ Good to Have — UptimeRobot already gives you the most important alert (site down). Sentry new-error alerts are the useful next step.

Essential Alerts for Food Delivery

AlertThresholdTool & ChannelWhy It Matters
Server DownUptime check fails 2 minutes in a rowUptimeRobot → Email + SMSUsers cannot access the site at all
New Error TypeFirst occurrence of any new unhandled errorSentry → EmailCatch new bugs the moment they happen
Error Spike> 10 errors in 5 minutesSentry → EmailSomething broke in production recently
Slow TransactionAny request > 2 000 msSentry Performance → EmailDetects performance regressions
Payment Failure Spike> 3 payment failures in 10 minutesCustom webhook → Slack/EmailRevenue impact; likely a gateway config issue

Custom Alert: Payment Failure Webhook

// src/services/alert.service.js const axios = require('axios'); async function sendSlackAlert(message) { if (!process.env.SLACK_WEBHOOK_URL) return; // skip if not configured await axios.post(process.env.SLACK_WEBHOOK_URL, { text: `🚨 *Food Delivery Alert*\n${message}` }); } // In payment controller — call when payment fails: // await sendSlackAlert(`Payment failed: orderId=${order.id}, amount=₹${amount}, error=${err.message}`);

4. Structured Logs

⭐ Good to Have

Making logs machine-readable and searchable

⭐ Good to Have — Structured logging is the professional standard. console.log is acceptable for a passing grade; JSON logs with a library like pino elevate the project.

Structured logging means writing logs as JSON objects rather than plain text strings. This makes them searchable, filterable, and parseable by log aggregation tools.

Plain Text vs Structured Log

Plain Text (Minimal)Structured JSON (Good to Have)
console.log('Order placed by user 42 for restaurant 7, total: 520') {"event":"order_placed","user_id":42,"restaurant_id":7,"total":520,"order_id":"ORD-001","ts":"2024-03-15T14:22:10Z"}

With structured logs, you can query: "Show all orders placed by user 42 in the last hour" or "Show all failed orders for restaurant 7" — impossible with plain text.

Setup: pino Logger (Node.js)

npm install pino pino-pretty pino-http // src/config/logger.js const pino = require('pino'); const logger = pino({ level: process.env.LOG_LEVEL || 'info', // In development: pretty-print; in production: raw JSON for log aggregators transport: process.env.NODE_ENV === 'development' ? { target: 'pino-pretty', options: { colorize: true } } : undefined, }); module.exports = logger; // src/app.js — HTTP request logging const pinoHttp = require('pino-http'); const logger = require('./config/logger'); app.use(pinoHttp({ logger }));

Using the Logger in Controllers

// src/controllers/order.controller.js const logger = require('../config/logger'); async function create(req, res) { const { restaurant_id, items } = req.body; const customerId = req.user.id; logger.info({ event: 'order_attempt', customerId, restaurant_id, itemCount: items.length }, 'Customer placing order'); try { const order = await orderService.placeOrder({ customerId, restaurant_id, items }); logger.info({ event: 'order_placed', orderId: order.id, customerId, restaurant_id, total: order.total }, 'Order placed successfully'); res.status(201).json({ data: order }); } catch (err) { logger.error({ event: 'order_failed', customerId, restaurant_id, error: err.message }, 'Order placement failed'); res.status(500).json({ error: 'ORDER_FAILED' }); } }
Log Retention

Keep logs for at least 30 days. On Render, logs are only kept for 7 days on the free tier. For longer retention, forward logs to Logtail or Papertrail — both have generous free tiers. Add LOGTAIL_SOURCE_TOKEN as an env variable and use @logtail/pino.
