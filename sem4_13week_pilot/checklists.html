Phase Checklists

One checklist per phase. Faculty uses these to approve team progression. Consultant uses the same list for spot checks. A team cannot progress to the next phase until all Gate items are checked.

5 Phases Pass/Fail Gates Faculty + Consultant
How to use: Items marked Gate are hard requirements — teams cannot move forward without them. Items without a gate are important but not blocking. Faculty should sign the gate at the bottom of each checklist. Consultant spot-checks randomly.

Phase 1 · Weeks 1–2

Scope & Requirements Gate

Completed after Week 2. Faculty approves before teams start design. Consultant spot-checks 2–3 teams.

Student Deliverables
  • Git repository created with standard folder structure Gate
  • README.md committed with project name and team members
  • Dockerfile stub committed (even if empty)
  • docker-compose.yml stub committed Gate
  • .gitignore includes node_modules, .env, build artifacts
  • T1 (scope.md) committed — all sections filled Gate
  • Problem statement is specific (not generic)
  • Scope clearly distinguishes in-scope from out-of-scope Gate
  • Risk register has at least 3 risks with mitigations
  • Tech stack and Docker service map defined
  • T2 (frd.md) committed — FRD, user stories, use cases Gate
  • Minimum 10 user stories, each with acceptance criteria Gate
  • At least 3 use cases written with main + alt flows
  • Test case headers defined (TC IDs linked to FRs)
Faculty & Consultant Checks
  • Scope is realistic for 13 weeks (not too big, not trivial) Faculty
  • Problem statement reflects a genuine problem, not a generic "we build a system"
  • User stories use correct format (As a / I want / So that)
  • FRs are specific — no vague "the system shall be user-friendly"
  • Docker service map matches the stated tech stack

Phase 2 · Weeks 3–4

Design Gate — Must Be Signed Before Sprint 1

Completed end of Week 4. No team starts coding until faculty signs this gate. Consultant reviews all RTMs.

T3 Design Doc Sections
  • Section 1: Introduction — purpose, scope, definitions
  • Section 2: System Overview — description + context diagram
  • Section 3: Architecture — pattern justified, component table Gate
  • Section 4: Docker service map — all containers defined Gate
  • Section 5: ER Diagram — all FRD entities, PKs, FKs, relationships Gate
  • Section 6: Level-1 DFD — main data flows visible
  • Section 7: Class diagram — key classes with attributes + methods
  • Section 8: Sequence diagrams — at least 2 key flows
  • Section 9: UI wireframes — at least 3 key screens
  • Section 10: RTM — every FR from T2 mapped to design Gate
  • All diagrams committed to docs/images/
  • design.md (T3) committed to repo
Quality Checks
  • ER diagram is consistent with FRD (every entity exists in FRD) Consultant
  • Docker service map is consistent with tech stack in T1
  • RTM covers 100% of FRs (no FR without a design component) Consultant
    What this means: The Requirements Traceability Matrix (RTM) in T3 has one row per functional requirement. Each row links a specific FR (e.g. FR-05: User can reset password) to the design component that implements it, the database table(s) it touches, the class(es) that own the logic, and the test case(s) that prove it works.
    Why it matters: If FR-05 has no row in the RTM, it means nobody designed how to build it, nobody knows which DB table supports it, and nobody wrote a test for it — so it will almost certainly be missed during coding. "100% of FRs" = every single FR from T2 has a corresponding row in the RTM. No requirement should be invisible to design.
  • Sequence diagrams show actual component names, not generic boxes
  • Architecture choice is justified (not just "we chose MVC")

Phase 3 · Weeks 5–10

Sprint Gate — Repeated Each Sprint (×3)

Applied at end of Sprint 1 (W6), Sprint 2 (W8), and Sprint 3 (W10). Faculty runs end-of-sprint demo. Consultant attends 1–2 demos per sprint cycle.

End-of-Sprint Requirements
  • Sprint plan (T4) committed to docs/sprints/sprint[N].md
  • All backlog tasks updated (Done / In Progress / Carried Over)
  • Weekly status reports submitted both mid and end of sprint
  • App runs via docker compose up — no manual steps Gate
  • Sprint goal user flow demonstrated live (not screenshots) Gate
  • No hardcoded credentials in committed code Gate
  • Regular commits throughout sprint (not one big commit on demo day)
  • Blockers log updated — all resolved or have action plans
  • Sprint retrospective filled in

Phase 4 · Weeks 11–12

Hardening & Documentation Gate

Completed end of Week 12. Docker deployment verified. All docs complete. Dry run demo done. Consultant's second session (Docker deployment) in Week 11.

Hardening Checklist
  • Code freeze — no new features after Week 10 Gate
  • App tested on a second machine (not just developer's machine) Gate
  • README deployment guide is complete and accurate Gate
  • .env.example committed with all required variables
  • Real .env is in .gitignore (never committed) Gate
  • All test results recorded in test-plan.md
  • Known issues documented in T5
Documentation Checklist (by Week 12)
  • T5 (final-report.md) all sections filled Gate
  • All diagrams updated to match final code
  • Individual contribution matrix filled in
  • Lessons learned written (not generic)
  • Dry run demo done with faculty feedback received
Week 11: Consultant Docker Session
Topics: docker compose in production, env_file vs environment, volumes and data persistence, reading container logs, common deployment issues and fixes.

Phase 5 · Week 13

Final Demo & Submission Checklist

Demo day. Panel includes faculty + HOD (optional external evaluator). Git repo URL is the final submission — no other format.

Before Demo (morning of Week 13)
  • Git repo link ready to share with panel Gate
  • App tested one final time: docker compose up on demo machine Gate
  • Final Submission Checklist in T5 completed and committed Gate
  • Presentation deck ready — format and duration at faculty's discretion
  • Demo flow rehearsed by at least 2 team members
  • Test credentials noted (e.g. test@demo.com / Demo@1234)
  • Retrospective committed to repo
During Demo — ~1 hour
Pacing and format is entirely at the college and faculty's discretion.
  • Problem statement introduced
  • Live demo of core user flow Gate
  • Architecture diagram shown and explained
  • Lessons learned shared
  • Q&A — team can answer technical questions
