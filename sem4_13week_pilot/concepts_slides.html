01 · Foundation

What is Software Engineering?

Definition: Software engineering is the disciplined way of building software — not just writing code, but planning it, designing it, testing it, and maintaining it so a team can deliver something reliable, on time, that other people can use and improve.
Example
Anyone can write a script that checks one password. Software engineering is what turns that into a real login system: it works for 500 students at once, stores passwords safely, locks out attackers, is tested, runs on any machine, and a teammate can understand your code six months later.
02 · Process

What is SDLC?

SDLC = Software Development Life Cycle. The ordered set of phases every project passes through, from idea to shipped product. It stops teams from jumping straight to coding and getting lost.
The 5 phases
  1. Requirements — what to build
  2. Design — how to build it
  3. Build (code) — actually build it
  4. Test — check it works
  5. Deploy & Maintain — ship & keep alive
03 · Requirements

Before You Build: 5 Questions

Every requirements artifact answers one plain question about the feature. Get these five right and you know what to build, for whom, and how you'll prove it works — before writing any code.
The five questions — Login feature

03 · Requirements

Functional Requirements Document (FRD)

Definition: A numbered list of exactly what the system must do — each item is testable. Functional = features (what it does). Non-functional = qualities (how well: speed, security).
Example
IDRequirement
FR-01A user shall log in with a registered email and password.
FR-02The system shall lock the account for 15 min after 5 wrong attempts.
FR-03On a correct password, the system shall start a session and open the dashboard.

Non-functional: login must respond in under 2 seconds; passwords must be stored hashed — never as plain text.

03 · Requirements

User Stories

Definition: A requirement written from the user's point of view in one plain sentence, using the template:
As a <role>, I want <goal>, so that <benefit>.
Each story gets acceptance criteria — the conditions that prove it's done.
Example

As a student, I want to log in with my email and password, so that I can securely access my courses.

Acceptance criteria: wrong password shows "Invalid email or password" · 5 failed tries locks the account for 15 min · a correct login opens the dashboard.

03 · Requirements

Use Cases

Definition: A step-by-step description of how an actor interacts with the system to achieve a goal — including what happens when things go wrong (alternate flows). More detailed than a user story.
Example — "Log in"

Actor: Student  ·  Pre-condition: student has a registered account
Main flow: 1) student enters email + password → 2) clicks Login → 3) system checks the password → 4) system creates a session → 5) dashboard is shown.
Alternate flow: at step 3, if the password is wrong → system shows "Invalid email or password" and stays on the login page.

Definition: A concrete check with input → expected output that proves a requirement works. Every FR should trace to at least one test case.

Architecture

Definition: The high-level shape of the system — its big parts and how they talk. You choose a proven pattern (e.g. Client–Server, MVC, Microservices) and justify why.

06 · Build

Coding in a Dev Environment

Definition: You write code in an IDE (Integrated Development Environment) — an editor plus tools that help you write, run, and debug. VS Code is the popular free choice.
Example — writing the login endpoint in VS Code 
def login(email, password):
    user = db.find_user(email)                  # FR-01
    if not user or user.is_locked:              # FR-02
        return error("Invalid email or password")
    if not bcrypt.verify(password, user.password_hash):
        user.failed_attempts += 1               # FR-02
        return error("Invalid email or password")
    session = create_session(user.id)           # FR-03
    return ok({ "token": session.token })

07 · Deploy

Docker for Deployment

Definition: Docker packs your app plus everything it needs (runtime, libraries, config) into a container — a box that runs the same on any machine. No more "but it works on my laptop."
Key words

Dockerfile = recipe to build one container.
Image = the built package.
Container = a running copy of the image.
docker-compose.yml = runs many containers together (app + db + email).

Example — run the whole app 
docker compose up

Starts web, api and db together. Your final demo is exactly this — one command on a fresh machine.

08 · Collaborate

GitHub for Code + Documentation

Definition: Git tracks every change to your files; GitHub stores that history online so the whole team shares one source of truth. Code and docs live together — no email, no WhatsApp submissions.
Example — your standard repo structure 
your-project-name/
├── README.md            # how to run the project
├── Dockerfile           # app container definition
├── docker-compose.yml   # all services (app + db)
├── src/                 # all application source code
└── docs/                # all project documentation
    ├── scope.md         # T1 — Charter & Scope
    ├── frd.md           # T2 — Requirements
    ├── design.md        # T3 — ER, class, sequence...
    ├── test-plan.md     # test cases & results
    └── final-report.md  # T5 — Final Product Doc

Bonus: GitHub renders Mermaid diagrams automatically — commit the .md and your ER/class/sequence diagrams just appear.
