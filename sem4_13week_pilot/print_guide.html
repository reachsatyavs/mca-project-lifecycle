BIMA — Bangalore Integrated Management Academy · MCA Sem 4

Word & Print
Document Guide

Every template (T1–T5) has a corresponding Word/PDF printout students submit physically. This page defines the exact sections, cover page format, and page limits for each document. Git repo remains the primary record — print is the formal submission.

5 Documents BIMA Cover Page Standard Print = formal submission Git = primary record
Common cover page format: Every printed document must use the BIMA standard cover page shown in each section below. The Git repo URL must appear on the cover page so the evaluator can cross-reference the digital version. Font: Arial or Times New Roman, 12pt body, 1.5 line spacing, 1-inch margins.

📄 D1 — Project Charter & Scope
Due: Week 1, Friday
Project Charter & Scope Document
Based on T1 — first document submitted for faculty approval
Est. 4–6 pages T1 template → Word Faculty signs cover page
Open T1 Template →
Document Title for Cover Page
"Project Charter & Scope Document — v1.0"
  • C
    Cover Page
    BIMA standard cover page — institution, programme, team, mentor, Git URL, date
    1 page
  • ToC
    Table of Contents
    Auto-generated in Word; section numbers and page numbers
    1 page
  • 1
    1. Project Identification
    Problem statement (3–5 sentences) · Proposed solution · Stakeholder table (role, description, need)
    ~1 page
  • 2
    2. Objectives & Success Criteria
    3–5 measurable objectives in "By Week 13, the system will…" format · Definition of Done
    ~0.5 page
  • 3
    3. Project Scope
    Two-column table: In-Scope (what will be built) vs Out-of-Scope (explicit exclusions)
    ~1 page
  • 4
    4. Constraints, Assumptions & Risk Register
    Constraints table · Assumptions list · Risk table (risk, likelihood, impact, mitigation)
    ~1 page
  • 5
    5. Technology Stack & Docker Architecture
    Tech stack table (layer, technology, reason) · Docker service map (container, image, port, purpose)
    ~0.5 page
  • 6
    6. Git Repository Details
    Repo URL · Folder structure diagram · Branching strategy · Team commit commitment
    ~0.5 page
  • S
    Signatures Page
    Team Lead signature · All members sign · Faculty Mentor sign-off with date
    1 page
Print tips: Scope table should be clearly readable — use a 2-column bordered table, not a bulleted list. Risk register: keep to 5–8 rows. Docker service map: 4 columns, bordered. Max 6 pages total — do not pad.
📄 D2 — Functional Requirements Document (FRD)
Due: Week 2, Friday
Functional Requirements Document
Based on T2 — defines what the system must do; baseline for design and testing
Est. 6–10 pages T2 template → Word Faculty approves before design starts
Open T2 Template →
Document Title for Cover Page
"Functional Requirements Document (FRD) — v1.0"
  • C
    Cover Page + Revision History
    BIMA cover page · Revision table (version, author, description, date)
    1 page
  • ToC
    Table of Contents
    Auto-generated
    1 page
  • 1
    1. Introduction
    Purpose of this document · Scope (what this FRD covers) · Definitions & abbreviations table
    ~0.5 page
  • 2
    2. Functional Requirements
    Numbered table: FR ID · Requirement description · Priority (High/Medium/Low) · Related user story
    1–2 pages
  • 3
    3. Non-Functional Requirements
    NFR table: NFR ID · Category · Requirement · Priority
    ~0.5 page
  • 4
    4. User Stories
    Table: Story ID · As a… · I want… · So that… · Priority · Acceptance Criteria. Minimum 10 stories.
    2–3 pages
  • 5
    5. Use Cases
    Minimum 3. Each: UC ID, Actor, Pre-condition, Main Flow (numbered steps), Alternate Flow, Post-condition
    2–3 pages
  • 6
    6. Test Case Headers
    Table: TC ID · Description · Related FR · Related US · Test Type (Positive/Negative/Integration)
    1 page
  • S
    Signatures Page
    Team Lead + Faculty sign-off
    1 page
Print tips: User stories table should use landscape orientation if needed — 5 columns is a lot for portrait. Acceptance criteria column: keep to 1–2 lines max in the table; detailed criteria go in the system. Use case flows: numbered list, not a paragraph. Do not print the interactive HTML form elements — fill first, then paste into Word.

Due: Week 4, Friday (Faculty Sign-off Gate)
Software Design Document (SDD)
Based on T3 — the most diagram-heavy document; includes all architectural decisions
Est. 10–15 pages T3 template → Word Diagrams as embedded images Faculty gate sign-off on last page
Open T3 Template →
Document Title for Cover Page
"Software Design Document (SDD) — v1.0"
  • C
    Cover Page + Revision History
    BIMA cover page · Revision table (version, author, change description, date)
    1 page
  • ToC
    Table of Contents + List of Figures
    Auto-generated in Word; include figure/diagram list
    1 page
  • 1
    1. Introduction
    Purpose · Scope · Definitions & abbreviations · References (FRD version, T1 scope doc)
    ~0.5 page
  • 2
    2. System Overview
    High-level description (1 paragraph) · System Context Diagram (Figure 1 — full width)
    1 page
  • 3
    3. System Architecture
    Pattern chosen + 2-line justification · Component table · Architecture Diagram (Figure 2)
    1–2 pages
  • 4
    4. Docker Architecture & Deployment Map
    Container services table (name, image, port, volumes, env vars) · docker-compose.yml snippet (code block)
    1 page
  • 5
    5. Data Design — ER Diagram
    Entity table (attributes, PK, FK) · ER Diagram (Figure 3 — full page recommended) · Relationships summary
    1–2 pages
  • 6
    6. Data Flow Diagram — Level 1
    DFD Level-1 diagram (Figure 4) · 3–5 sentence description of main data flows
    1 page
  • 7
    7. Class Diagram
    Key classes table (attributes, methods, relationships) · Class Diagram (Figure 5)
    1–2 pages
  • 8
    8. Sequence Diagrams
    Flow 1 — Sequence Diagram (Figure 6) + description · Flow 2 — Sequence Diagram (Figure 7) + description
    2 pages
  • 9
    9. User Interface Wireframes
    3 key screens — each as a figure (Figures 8–10) with a brief caption explaining key elements
    1–2 pages
  • 10
    10. Requirements Traceability Matrix (RTM)
    Table: FR ID · Requirement · Design Component · DB Table · Class(es) · TC ID(s) — all FRs must appear
    1 page
  • S
    Design Gate Sign-off
    Faculty sign-off with date — "Design approved. Team authorised to begin Sprint 1."
    Part of last page
Print tips: Every diagram must be clear and legible — if it's too small to read in print, put it on its own page. Caption every figure: "Figure 1: System Context Diagram — [Project Name]". ER diagram especially must be on a full page (landscape if needed). Do NOT reduce font below 10pt to fit content — reduce content instead. RTM table: use condensed font (Arial Narrow 10pt) or landscape orientation.

Due: End of Sprint 3 (Week 10)
Combined Sprint Log
Based on T4 — one document covering all 3 sprints; submitted after Sprint 3 demo
Est. 10–12 pages 3 sprints in one doc T4 template (×3) → Word
Open T4 Template →
Document Title for Cover Page
"Combined Sprint Log — Sprints 1, 2 & 3"
  • C
    Cover Page
    BIMA cover page with sprint date range (Week 5 – Week 10)
    1 page
  • S
    Sprint Summary Overview
    One-page table: Sprint | Goal | Dates | Goal Achieved? | Tasks Done / Total | Key Demo Outcome
    1 page
  • 1
    Sprint 1 — Weeks 5–6
    Sprint goal · Backlog table (tasks, owner, estimate, status, DoD) · Blockers log · Weekly status summaries · Retrospective (well/wrong/change) · Demo sign-off by faculty
    3–4 pages
  • 2
    Sprint 2 — Weeks 7–8
    Same structure as Sprint 1 above
    3–4 pages
  • 3
    Sprint 3 — Weeks 9–10
    Same structure + code freeze declaration + test execution note
    3–4 pages
  • S
    Faculty Sign-off (end of doc)
    Faculty signs confirming all 3 sprint demos were conducted and gates were passed
    Part of last page
Print tips: Backlog table can be printed in landscape. Each sprint starts on a new page (use Word page break). Retrospective: use a 3-column table (Went Well / Went Wrong / Will Change). Do not print screenshots of code — link to Git commits instead. Blockers log: only resolved blockers need be in the print version — active blockers are in the Git repo status files.
📄 D5 — Final Product Documentation
Due: Week 13, Demo Day
Final Product Documentation
Based on T5 — the complete project record; the capstone submission document for BIMA MCA
Est. 15–20 pages T5 template → Word Bound + submitted on demo day Faculty sign
Open T5 Template →
Document Title for Cover Page
"Final Product Documentation — [Project Name] — MCA Sem 4, BIMA (2025–26)"
  • C
    Cover Page
    BIMA cover page · All team members · Faculty mentor · Git repo URL · Submission date
    1 page
  • ToC
    Table of Contents + List of Figures
    Auto-generated; include all diagrams in the figure list
    1 page
  • A
    Abstract
    150–200 words: problem, solution, technologies, key outcomes. Standalone paragraph — evaluator reads this first.
    ~0.5 page
  • 1
    1. Project Scope — Achieved vs Planned
    Table: scope items from D1 vs actual completion status · Requirements summary (FR count, US count, test count)
    1 page
  • 2
    2. Design Summary
    Final architecture pattern · Final tech stack table · Diagram index (figure name, file path, updated?) · Any design changes from D3
    1–2 pages
  • 3
    3. Deployment Guide
    Prerequisites · Step-by-step clone & run instructions · docker-compose.yml snippet · .env.example · Default test credentials for demo
    1–2 pages
  • 4
    4. Test Results Summary
    Pass/fail counts (total/pass/fail/skip) · TC results table (ID, description, result, notes) · Reference to full test-plan.md in repo
    1–2 pages
  • 5
    5. Sprint Summary
    3-column table: Sprint 1 / Sprint 2 / Sprint 3 — goal, achieved?, key outcomes for each
    1 page
  • 6
    6. Known Issues & Limitations
    Table: issue, impact, workaround. Be honest — evaluators value transparency.
    ~0.5 page
  • 7
    7. Lessons Learned
    3-column table: What went well · What went wrong · What we'd do differently. Plus: key technical learnings paragraph.
    1 page
  • 8
    8. Individual Contribution Matrix
    Table: member name · modules/features owned · documents authored · Git contribution % — must be specific and verifiable
    ~0.5 page
  • 9
    9. Final Submission Checklist
    Printed version of the checklist from T5 — all items checked, team lead signed
    1 page
  • S
    Sign-off Page
    Team Lead signature · All team members · Faculty Mentor
    1 page
Print tips: This is a bound document — use a spiral or comb binding. All diagrams from D3 must reappear here (updated to reflect final code, not initial design). Deployment guide: someone who has never seen your project must be able to run it using only this guide. Contribution matrix must reference actual Git commits — vague descriptions will not be accepted. Abstract goes on its own page (no heading above it, centered at top of page 3).
