Maintenance Guide

Enabling future maintainers to keep the system healthy

The Maintenance Guide is written for someone who didn't build the system but needs to maintain it. It covers routine operations, troubleshooting, and how to perform common changes.

Maintenance Guide Contents

SectionContents
System OverviewArchitecture diagram, component list, all URLs (production, staging, API docs, monitoring dashboards)
Access CredentialsWhere to find credentials (password manager / secrets vault — not in this doc!). Who has admin access.
Routine OperationsHow to: restart the server, run migrations, create admin user, restore from backup, view logs
MonitoringUptimeRobot dashboard URL, Sentry project URL, Render dashboard URL
Hotfix ProcessStep-by-step hotfix workflow (git branching + staging verification + production deploy)
Dependency UpdatesHow to run pip audit and npm audit; checklist for updating Express.js/React versions
Common IssuesKnown issues, their symptoms, and fixes: Render auto-sleep, DB connection pool, CORS errors
ContactsTeam members, faculty guide, hosting support links

Final README

The polished, complete face of the project's GitHub repository

Final README Checklist

  • ✅ Project title, tagline, and 1-paragraph description.
  • ✅ Live demo URL (production URL).
  • ✅ Screenshots of key screens (home, ordering flow, dashboard).
  • ✅ Feature list — what the system can do.
  • ✅ Tech stack — with version numbers.
  • ✅ Local development setup (docker-compose up).
  • ✅ Running tests (jest --cov=apps).
  • ✅ Links to all documents (SRS, TDD, SPMP PDFs in /docs).
  • ✅ Team members with roles.
  • ✅ Project status badges (CI status, coverage %, uptime).

Project Report

The comprehensive academic document summarising 4 semesters of work

The Final Project Report is the primary academic deliverable submitted for the MCA project. It is typically 80–120 pages and follows the university's prescribed format.

Recommended Chapter Structure

ChapterContentsApprox Pages
1. IntroductionBackground, problem statement, objectives, scope, report organisation5–8
2. Literature SurveyReview of existing food delivery systems (Swiggy, Zomato, DoorDash), similar academic projects, technologies reviewed8–12
3. System AnalysisRequirements (FR, NFR), Use Cases, Business Rules — condensed from SRS15–20
4. System DesignArchitecture, ER diagram, class diagrams, API design, UI wireframes — condensed from TDD15–20
5. ImplementationTech stack, module-by-module implementation details, key algorithms (order assignment, rating calculation), code snippets for complex logic20–25
6. TestingTest plan summary, unit test results, integration test results, E2E results, performance results, security scan results, UAT certificate15–20
7. DeploymentCI/CD pipeline overview, hosting architecture, production deployment steps, monitoring setup5–8
8. Conclusion & Future WorkWhat was achieved, what was learned, limitations, future enhancements (mobile app, AI recommendations, multi-city)5–8
ReferencesIEEE format references for all cited papers, websites, frameworks2–4
AppendicesFull SRS, SPMP, TDD; sample test cases; deployment guide; user manualVaries
Write Progressively

Don't write the project report in Semester 4 from scratch. Start a report template in Semester 1 and add to it every semester. Chapters 3 and 4 (Analysis and Design) are essentially the SRS and TDD converted to report format — you've already done most of the work.
