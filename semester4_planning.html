Maintenance Plan

How the system will be kept healthy after production launch

A Maintenance Plan defines procedures for handling bugs, patches, and updates after the system is live. Even a student project needs a maintenance plan for the duration of the Semester 4 evaluation period.

Maintenance Categories

TypeDescriptionExample for Food DeliveryResponse Time
CorrectiveFixing bugs found in productionOrder status stuck at "Preparing" due to partner app bugP1: 24hrs. P2: 1 week
AdaptiveChanges to keep up with environmentRazorpay API version update requires library upgradeWithin 2 weeks of change
PerfectivePerformance or UX improvementsReduce restaurant listing page load from 1.2s to 0.4s with cachingEnhancement log — next sprint
PreventivePreventing future failuresAdd DB index after load test reveals slow queryScheduled maintenance window

Maintenance Schedule

  • Daily: Monitor Sentry dashboard and UptimeRobot alerts.
  • Weekly: Review open bug list; triage new bugs from Sentry.
  • Monthly: Dependency security audit (pip audit); review database backup integrity.
  • Every 3 months: Review and update dependencies to latest stable versions.

Retrospective

Looking back on 4 semesters — what worked and what to improve

A retrospective is a structured reflection session run at the end of the project. The team honestly evaluates their process, tools, and teamwork to derive lessons for future projects.

Retrospective Format (4Ls)

CategoryQuestionExample Responses (Food Delivery Team)
Liked ❤️What worked really well?"GitHub Actions CI saved us 3 times from merging broken code." "Weekly standups kept everyone accountable." "Using Swagger made frontend integration much easier."
Learned 📚What did we learn?"Payment gateway integration is harder than it looks — start early." "Docker solved the 'works on my machine' problem completely." "User testing revealed bugs we never would have found internally."
Lacked 😟What was missing or insufficient?"We didn't start writing tests until Semester 3 — should have been Semester 2." "Our Semester 1 ER diagram had 3 missing tables discovered in Semester 2." "No on-call schedule — P1 bugs sat for 48hrs on weekends."
Longed For 🌟What do we wish we had?"A real mobile app." "Redis caching from the start." "More beta testers with diverse devices."

Key Lessons for Future MCA Projects

  1. Write tests from Semester 2 alongside the code — not after.
  2. Set up CI/CD in Semester 2 — not Semester 3.
  3. Define API contracts before writing frontend — it enables parallel development.
  4. Get real users to test early and often — at least once per semester.
  5. Document as you go — don't leave documentation for the final week.
  6. The Non-Goals document saved the project from three attempts at scope creep.
