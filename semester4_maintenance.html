1. Hotfixes

Emergency fixes for P1/P2 production bugs

A hotfix is an emergency code change deployed outside the normal sprint cycle to fix a critical production bug. The key constraint: speed + safety. Fix fast, but don't break anything else.

Hotfix Decision Matrix

Real Hotfixes in Food Delivery

These are plausible hotfixes a student team might encounter in Semester 4:

  • HF-001: Razorpay updated their checkout.js CDN URL — old URL returns 404. Fix: update CDN URL in base template.
  • HF-002: Express.js security update released (Express.js 4.2.10 patches CVE-XXXX). Fix: update package.json and redeploy.
  • HF-003: Render free tier DB connection limit hit during demo day (25 connections). Fix: reduce database connection pool size in settings.

2. Version Updates

Keeping dependencies secure and up-to-date

Dependencies that were current in Semester 1 may have known security vulnerabilities by Semester 4. Keeping dependencies updated is a basic security hygiene responsibility.

Dependency Audit Commands

# Python backend — check for known vulnerabilities pip audit # Sample output: # Found 1 known vulnerability in 1 package: # name: Pillow, version: 9.3.0 # id: PYSEC-2023-xxx, fix: upgrade to >= 10.0.1 pip install --upgrade Pillow pip freeze > package.json # ---- # Node.js frontend npm audit # Sample output: # 2 high severity vulnerabilities # Run `npm audit fix` to fix them npm audit fix

Safe Update Process

  1. Run pip audit / npm audit to identify vulnerabilities.
  2. Update vulnerable package to patched version.
  3. Run full test suite — ensure nothing broke.
  4. Deploy to staging; smoke test.
  5. Deploy to production via hotfix branch.
  6. Document the update in the maintenance log.

3. Post-Deployment Review

Assessing production performance 2–4 weeks after go-live

The Post-Deployment Review (PDR) is conducted 2–4 weeks after the production launch. It answers: Is the system performing as expected? Are users able to use it? Are the NFRs being met in production?

PDR Agenda

  • Performance metrics review: Compare Sentry/UptimeRobot/Render data against NFR targets (see SLA table in Monitoring Sem 4).
  • Error review: What are the top 5 errors in Sentry this month? Are they fixed or known?
  • User feedback review: Any complaints from beta testers still using the system?
  • Cost review: Is the system within free-tier limits? Any risk of incurring charges?
  • Security review: Any new CVEs in our dependencies since go-live?
  • Decisions: What needs to be fixed? What should go into the Enhancement Log?

4. Enhancement Log

Recording ideas for future versions — beyond the MCA project

The Enhancement Log is a backlog of features and improvements that were explicitly out of scope for the 4-semester project but are worth building in a future version. It shows the evaluators that you understand the product's growth potential.

Include in Project Report

Include the Enhancement Log in Chapter 8 (Conclusion & Future Work) of your project report. It demonstrates product thinking — understanding the product's roadmap beyond the current semester's scope. Evaluators appreciate when students show awareness of what a production-grade system would require.
