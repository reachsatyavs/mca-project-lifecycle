User stories capture requirements from the end user's perspective in plain, non-technical language. Each story is small enough to be built in one sprint and carries explicit acceptance criteria that drive test cases.

Format & Writing Tips

How to write user stories that are actually useful

User Story Format

As a [role], I want to [action], so that [benefit].

INVEST Checklist — Every Story Should Be:

LetterMeaningExample Check
IndependentCan be built without another story being done first"Login" does not depend on "View Order History"
NegotiableNot a fixed contract — details can changePayment method can start with COD only
ValuableDelivers value to the user or businessEvery story answers "why does this matter?"
EstimableTeam can assign story points to itTeam agrees it's a 5-point story
SmallFits in one sprint (2 weeks)Split "full order flow" into 3 separate stories
TestableHas acceptance criteria that can be verified"Cart shows item count badge" — easy to test

Each story should also have Acceptance Criteria — specific conditions that must be true for the story to be "done". These become the basis for your test cases in Semester 2.

Customer Stories

Stories from the perspective of a food-ordering customer

Customer · US-C01
As a customer, I want to register an account so that I can place orders and track my order history.
Acceptance Criteria
  • Registration form accepts name, email, phone, and password.
  • Password must be ≥ 8 characters with at least one number.
  • A verification email is sent; account is inactive until verified.
  • Duplicate email shows "Email already registered."
Customer · US-C02
As a customer, I want to log in with my email and password so that I can access my saved addresses and order history.
Acceptance Criteria
  • Valid credentials redirect to the home page; JWT is set.
  • Invalid credentials show "Invalid email or password" — no redirect.
  • Session persists across page reload via refresh token.
Customer · US-C03
As a customer, I want to search for restaurants by cuisine type so that I can quickly find food that matches my preference.
Acceptance Criteria
  • A search bar and cuisine filter are visible on the home page.
  • Selecting "Chinese" shows only approved Chinese restaurants.
  • No matches shows a friendly "No results" message.
  • Results appear within 2 seconds.
Customer · US-C04
As a customer, I want to add multiple items to my cart from a restaurant so that I can order everything I need in one transaction.
Acceptance Criteria
  • Each menu item has an "Add to Cart" button with a quantity stepper.
  • Cart icon shows live item count badge.
  • Adding items from a second restaurant shows a warning dialog: "Clear cart?"
  • Cart persists across page refresh (localStorage or session).
Customer · US-C05
As a customer, I want to pay online for my order so that I don't need to carry cash.
Acceptance Criteria
  • Checkout page shows itemised breakdown (subtotal, delivery fee, taxes, total).
  • Customer can choose Credit Card, Debit Card, UPI, or COD.
  • Successful payment creates the order and redirects to tracking page.
  • Failed payment shows an error with a retry option — order is NOT created.
Customer · US-C06
As a customer, I want to track my order in real-time after placing it so that I know when to expect my food.
Acceptance Criteria
  • After placing an order, customer is redirected to an order tracking page.
  • A visual timeline shows: Placed → Accepted → Preparing → Out for Delivery → Delivered.
  • Status auto-refreshes every 15 seconds without full page reload.
  • Email/push notification sent when status changes.
Customer · US-C07
As a customer, I want to rate and review a restaurant after delivery so that other users can make informed choices.
Acceptance Criteria
  • "Rate Your Order" prompt appears after order status = DELIVERED.
  • Customer can give 1–5 stars and add an optional text comment (max 500 chars).
  • Each customer can only review a specific order once.
  • Restaurant's average rating updates within 1 minute of submission.
Customer · US-C08
As a customer, I want to view my order history so that I can reorder a favourite meal quickly.
Acceptance Criteria
  • Order history lists all past orders sorted by date (newest first).
  • Each entry shows restaurant name, items, total, and status.
  • "Reorder" button re-adds the same items to the cart in one click.
  • Customer can view a full receipt/invoice for each order.

Restaurant Owner Stories

Stories from the perspective of a registered restaurant owner

Restaurant Owner · US-O01
As a restaurant owner, I want to register my restaurant on the platform so that customers in my area can discover and order from me.
Acceptance Criteria
  • Registration form accepts name, address, cuisine type, opening hours, FSSAI licence number.
  • Owner is notified by email when registration is submitted.
  • Restaurant is NOT visible to customers until admin approves it.
  • Owner receives email notification upon admin approval or rejection (with reason).
Restaurant Owner · US-O02
As a restaurant owner, I want to manage my menu online so that I can add seasonal dishes and update prices without contacting a developer.
Acceptance Criteria
  • Dashboard has a "Menu" tab with a full list of items.
  • Owner can add a new item with name, price, category, photo, and veg/non-veg flag.
  • Owner can toggle item availability without deleting it.
  • Changes appear on the customer-facing menu within 30 seconds.
Restaurant Owner · US-O03
As a restaurant owner, I want to receive and manage incoming orders instantly so that I can begin preparation without delay.
Acceptance Criteria
  • New order triggers a visual and audio notification on the dashboard.
  • Owner can Accept or Reject within 5 minutes; timeout = auto-reject.
  • Accepted orders show items, quantity, delivery address, and payment mode.
  • Rejected orders notify the customer with a reason.

Delivery Partner Stories

Stories from the perspective of a delivery partner

Delivery Partner · US-P01
As a delivery partner, I want to go online/offline on the platform so that I only receive orders when I am available.
Acceptance Criteria
  • A prominent toggle on the partner portal shows Online / Offline status.
  • When offline, no new orders are assigned to the partner.
  • Going offline during an active delivery is not allowed until the order is delivered.
Delivery Partner · US-P02
As a delivery partner, I want to see assigned orders with restaurant and customer details so that I know exactly where to go for pickup and delivery.
Acceptance Criteria
  • Partner portal shows: restaurant name, address, item list, customer address.
  • Partner can mark order as "Picked Up" once collected from restaurant.
  • Partner can mark order as "Delivered" upon successful handover.
  • Customer receives push/email notification on each status change.

Admin Stories

Stories from the perspective of the platform administrator

Admin · US-A01
As an admin, I want to approve or reject new restaurant registrations so that only legitimate, verified restaurants are listed on the platform.
Acceptance Criteria
  • Admin dashboard shows a list of pending restaurant applications.
  • Admin can view submitted details and approve or reject with a reason.
  • Restaurant owner is notified by email upon approval or rejection.
Admin · US-A02
As an admin, I want to view platform-wide analytics so that I can monitor the health and growth of the platform.
Acceptance Criteria
  • Dashboard shows: total orders today, total revenue today, active restaurants, active users.
  • Trend charts show daily order volume for the past 30 days.
  • Admin can filter statistics by date range.
Admin · US-A03
As an admin, I want to suspend or ban a user account so that I can remove bad actors from the platform.
Acceptance Criteria
  • Admin can search for any user by name or email.
  • Admin can suspend (temporary) or ban (permanent) a user with a reason.
  • Suspended/banned users see a clear message when they try to log in.
  • Suspension can be reversed by admin.
User Stories → Sprint Backlog

These stories form the Product Backlog. In Sprint Planning (Semester 2), the team selects the top-priority stories from the backlog and commits to completing them within the 2-week sprint. The Sprint Backlog is the subset of stories selected for one sprint.
