Production Deployment

Going live for real — the final step of the 4-semester journey

Production Deployment Steps

  1. Final code freeze: Merge all approved PRs. No new features after code freeze — only bug fixes from this point.
  2. Run full regression suite on staging: all tests must pass.
  3. Run Go-Live Checklist (see Planning Sem 3).
  4. Database backup: Take a manual backup of the staging DB before production deploy.
  5. Set production environment variables: DATABASE_URL (prod DB), SENTRY_DSN (prod project), ALLOWED_HOSTS.
  6. Deploy to production Render service.
  7. Run database migrations: python manage.py migrate on the production instance.
  8. Smoke test production URL: Manually verify 5 critical flows work on the live site.
  9. Announce go-live to faculty guide. Send public URL.

Blue-Green Deployment (Optional, Advanced)

For zero-downtime deployment, consider a Blue-Green strategy: run two identical environments (Blue = current production, Green = new version). Switch traffic to Green once verified, keep Blue as instant rollback. Render supports this via preview environments.

CD Pipeline — Final

The complete Continuous Deployment flow with manual approval gate

# .github/workflows/cd-production.yml name: CD — Deploy to Production on: workflow_dispatch: # Manual trigger only (not auto) inputs: confirm: description: 'Type "deploy" to confirm production deployment' required: true jobs: deploy-production: runs-on: ubuntu-latest if: github.event.inputs.confirm == 'deploy' environment: production # Requires approval in GitHub Environments steps: - name: Run final test suite run: jest --tb=short - name: Deploy to Render Production run: | curl -X POST "${{ secrets.RENDER_PRODUCTION_DEPLOY_HOOK }}" - name: Notify team run: echo "Production deployment triggered. Monitor at Render dashboard."

The workflow_dispatch trigger means this CD pipeline only runs when a team member manually triggers it — not on every push. The environment: production setting can require a second-person approval in GitHub's environment protection rules.

Rollback Strategy

What to do when a production deployment causes a regression

A rollback reverts production to the last known-good state as quickly as possible. Every production deployment should have a tested rollback plan.

Rollback Options (Fastest to Slowest)

MethodTime to RollbackWhen to Use
Render: Deploy Previous Version~3 minutesCode issue — previous Docker image still available in Render
Git Revert + Redeploy~10 minutesCode issue — create a revert commit, push to main, CI/CD redeploys
Database Rollback (Migration Down)~20 minutesDB migration broke something — run python manage.py migrate myapp 0042
Full DB Restore from Backup~30–60 minutesLast resort — data corruption occurred

Test Your Rollback

Untested Rollbacks Fail

Run a mock rollback on staging before production go-live. Deploy a "bad" version to staging, then roll it back using option 1 above. If you've never practiced it, you will fumble when it matters most — during a real incident.
