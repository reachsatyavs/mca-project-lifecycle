Clinical Product Suite Architecture

A common platform for patient management, users, reports, licensing integration, upgrades, backups, and shared UI, consumed by multiple desktop healthcare products.

Executive Summary

Purpose of this draft

This is an early visualization draft for discussion, not a final design. It shows how the current Sonique desktop product could evolve into a broader clinical product suite.

1 Clarify the business vision for multiple clinical products.
2 Identify which capabilities should be common across products.
3 Collect decisions needed before detailed architecture and implementation.

Desktop Installation Model

Every product installer can bundle the Common Clinical Service. The first product installation creates the local platform. A second product installation detects the existing platform, checks compatibility, and prompts before any shared service upgrade.

Version and Upgrade Strategy

Since each product bundles the platform, version compatibility must be explicit. The installed Common Clinical Service should not be downgraded automatically. 

sequenceDiagram
  participant Installer as Product Installer
  participant VersionManager as update-core
  participant Service as Common Clinical Service
  participant Backup as backup-core
  participant DB as Shared DB

  Installer->>VersionManager: Check installed platform version
  VersionManager->>Service: GET /health/version
  Service-->>VersionManager: v1.2.0
  VersionManager->>VersionManager: Compare with required range
  VersionManager->>Backup: Create pre-upgrade backup
  Backup->>DB: Snapshot shared database
  VersionManager->>Service: Stop service safely
  VersionManager->>DB: Run migrations via db-core
  VersionManager->>Service: Start upgraded service
  Service-->>Installer: Healthy, v1.3.0

Questions We Have To Arrive At Answers

These questions are intentionally broad. The goal is to use the visualization above to collect product vision, business model, clinical workflow, deployment, and technology direction before moving into detailed design.

Product Suite Scope

  1. What are the first 2-3 products expected besides Sonique/Audiometry?
  2. Will each product be a separate desktop app, or should there also be one suite/launcher app?
  3. Should products be installable independently?
  4. Should all products share the same company branding, or can each product have its own branding?

Shared Patient Database

  1. Should one patient record be shared across all products?
  2. Do we need a universal patient ID generated by the system?
  3. Do clinics already have external patient IDs or hospital IDs that must be stored?
  4. Can the same patient visit multiple clinics/branches, or is patient data local to one clinic installation?
  5. Do we need duplicate patient detection?
  6. What minimum patient fields are mandatory?

Common Clinical Service

  1. Should the Common Clinical Service run silently in the background after installation?
  2. Should the Common Clinical Service start automatically when the computer starts?
  3. Should product apps connect to the Common Clinical Service using localhost on the same machine?
  4. Should future product apps on other clinic machines connect to one desktop/server machine over LAN?
  5. Should the Common Clinical Service expose REST APIs, or should we consider gRPC/WebSocket also?

Database, Backup, And Restore

  1. For the first version, should the shared local DB be SQLite or PostgreSQL?
  2. Is installation simplicity more important than multi-user capability for v1?
  3. Should backup/restore be handled centrally by the Common Clinical Service?
  4. Do we need automated scheduled backups?
  5. Should backups include all products' data together, or allow product-wise backup/export?

Users, Roles, And Security

  1. Do doctors/staff need login inside the desktop apps?
  2. What roles are expected, such as Admin, Doctor, Technician, Receptionist, or Report Viewer?
  3. Should each product enforce role-based access?
  4. Should the app support multiple doctors in the same clinic?
  5. Do we need audit logs?

Reports

  1. Should there be a common report framework used by all products?
  2. Should each product provide its own report template sections?
  3. Can one report combine results from multiple products?
  4. Should reports be PDF only, or also HTML, print, or image export?
  5. Do clinics need custom letterhead, logo, header, and footer?
  6. Should old generated reports be stored as immutable snapshots, or regenerated every time from latest data?

Product-Specific Modules

  1. For Audiometry, should AC/BC/FF/Speech remain fully inside the Sonique product module?
  2. Should product-specific test data live in the same shared DB, or should each product have its own DB?
  3. Should common reports be allowed to read product-specific data through registered report plugins?
  4. Should product modules register capabilities with the Common Clinical Service?

Hosted License Service

  1. Should desktop apps validate license directly, or should the Common Clinical Service validate once and share status with all products?
  2. Is license per clinic, per machine, per doctor, or per product?
  3. Should license allow feature flags such as Device Enabled, Reports Export, or Multi-user?
  4. Should the app work offline if the license server is unreachable?
  5. If offline use is allowed, how long should the offline grace period be?
  6. Should one license cover multiple products in the suite?
  7. Should license activation bind to machine ID?

Future Web Compatibility

  1. In future web mode, do we imagine cloud-hosted SaaS, customer-hosted server, or both?
  2. Should desktop and web share the same API contracts?
  3. Do we expect data sync between desktop offline installations and cloud later?
  4. Should we design for multi-tenant data from the beginning?

Repository And Package Strategy

  1. Do we prefer separate git repos for every core module, or one common platform repo initially?
  2. Are we comfortable using private npm packages?
  3. Where will code be hosted, for example GitHub private repos?
  4. Should each product have independent release/versioning?
  5. Should common modules follow semantic versioning, such as patient-core@1.2.0?

Settings, Clinics, Doctors, And Migration

  1. Should the Common Clinical Service also own common settings/preferences?
  2. Should clinic profile data be common, such as clinic name, address, logo, contact details, and registration number?
  3. Should doctors/consultants be modeled separately from app users?
  4. Should referring doctors be a separate common entity?
  5. Should appointments/visits be part of the common platform now, or later?
  6. Should every product create a common encounter or visit record before creating product-specific test records?
  7. Should reports be linked to visits/encounters, patients directly, or both?
  8. Should the system support import/export of patient data?
  9. Should the system support migration from the current Sonique SQLite database into the new shared database?

AI Opportunities

AI can be introduced in phases. The first phase should assist clinicians and internal teams without replacing clinical judgment. Later phases can use richer product data, report history, and support knowledge to improve productivity and decision support.

CClinic-Level AI Use Cases

AI features that directly help doctors, technicians, reception staff, and clinics.

  • Clinical note drafting: Generate draft notes from test results, observations, and selected templates.
  • Report summarization: Produce patient-friendly and doctor-friendly summaries from completed tests.
  • Abnormal result highlighting: Flag patterns that may need review, such as asymmetry or significant threshold changes.
  • Patient history assistant: Summarize previous visits, reports, and product-specific test history.
  • Voice-to-notes: Convert spoken observations into structured notes during or after a test.
  • Follow-up suggestions: Suggest possible follow-up actions based on configured clinic protocols.
  • Multilingual report explanation: Explain report findings in simple language for patients.

PPlatform / Company-Level AI Use Cases

AI features that help the company support products, customers, quality, and growth.

  • Support assistant: Answer customer questions using manuals, release notes, FAQs, and known troubleshooting steps.
  • Installation diagnostics: Analyze logs from installers and Common Clinical Service startup failures.
  • License and usage insights: Detect activation issues, expired licenses, and product usage trends.
  • Release impact assistant: Summarize what changed between platform/product versions and who may be affected.
  • Knowledge base generation: Convert support tickets and fixes into searchable help articles.
  • Product analytics: Identify commonly used workflows, underused modules, and feature adoption by product.
  • Internal engineering assistant: Help generate test cases, migration notes, API documentation, and support scripts.
