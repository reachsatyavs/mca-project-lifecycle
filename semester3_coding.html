Frontend Error States (Often Forgotten)

Always Handle These States
  • Loading state: Show skeleton/spinner while API is fetching.
  • Empty state: "No restaurants found" instead of blank page.
  • Error state: "Something went wrong, please try again" with retry button.
  • Offline state: Browser offline detection with user-friendly message.
  • 403/401: Redirect to login page (don't show blank screen).

2. Integration Layer

Connecting frontend to backend and all external services

Integration means ensuring that all parts of the system work together correctly — the frontend calls the right API endpoints, error responses from the backend are displayed properly on the frontend, and all external services (payment, email, image storage) work in the integrated system.

Frontend API Client (React / Axios)

// src/api/client.js import axios from 'axios'; const apiClient = axios.create({ baseURL: import.meta.env.VITE_API_BASE_URL, // from .env headers: { 'Content-Type': 'application/json' } }); // Attach JWT token to every request apiClient.interceptors.request.use(config => { const token = localStorage.getItem('accessToken'); if (token) config.headers.Authorization = `Bearer ${token}`; return config; }); // Handle 401 — redirect to login apiClient.interceptors.response.use( response => response, error => { if (error.response?.status === 401) { localStorage.removeItem('accessToken'); window.location.href = '/login'; } return Promise.reject(error); } ); export default apiClient;

CORS Configuration (Backend)

# settings/production.py CORS_ALLOWED_ORIGINS = [ "https://your-frontend.vercel.app", ] CORS_ALLOW_CREDENTIALS = True # Required for cookie-based auth

3. Feature Completion

Definition of Done for every feature

A feature is only "done" when it meets the Definition of Done (DoD) — not when a developer says "I'm done coding."

Definition of Done Checklist

  • Feature is implemented and matches acceptance criteria.
  • Unit tests written and passing (coverage maintained at ≥ 70%).
  • Integration test written for the main success and failure paths.
  • Code reviewed and approved by one other team member.
  • PR merged to develop branch (not just sitting in a local branch).
  • Feature is deployed to staging and manually smoke-tested.
  • Any new API endpoints are documented in Swagger.
  • No P1 or P2 bugs open against this feature.

4. Refactoring

Improving code quality without changing external behaviour

Refactoring is changing the internal structure of code to make it cleaner, more efficient, or more maintainable — without adding new features or fixing bugs. In Semester 3, you refactor code that was written quickly in Semester 2.

Common Refactoring Targets in Food Delivery

ProblemRefactoring
Order placement logic is all in the view (fat view anti-pattern)Extract to an OrderService class — view just calls service methods
Database queries duplicated across viewsExtract to a RestaurantRepository class or Manager
Magic numbers scattered in code (e.g., if total > 500)Move to named constants: FREE_DELIVERY_THRESHOLD = 500
N+1 query in restaurant listing (loads menu for each restaurant)Use prefetch_related('menuitem_set')
Password reset logic duplicated in two endpointsExtract to PasswordResetService
Refactoring Rule

Only refactor code that has unit tests. If you refactor without tests, you might silently break behaviour. The test suite is your safety net — it tells you instantly if a refactor broke something.
